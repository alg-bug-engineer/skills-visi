# 专家经验编写指南

本文说明交通专家在开发阶段如何编写经验，以及工程师如何把专家经验编译为运行时使用的 Skill Pack、rules、YAML 和脚本草稿。

## 总流程

1. 交通专家只维护 `路口专家经验规则.md`，每条经验使用稳定 ID 和固定字段。
2. 工程师运行专家经验编译器，生成结构化 IR 和 dry-run diff。
3. 工程师审核 diff，确认是否写回运行时文件。
4. `--apply` 写回 Markdown/YAML/source map；Python 只生成 `scripts/generated/` 草稿。
5. 运行检查和测试，通过后再提交。

## 编写条目格式

专家经验必须写在 `路口专家经验规则.md`，按“阶段、类型、现象、判别依据、适用边界、不要误判为、建议生成、运行时意图”组织。

示例：

```markdown
### EXP-DIAG-010 短间距串联路口

- 阶段：problem_diagnosis
- 类型：checklist_item, threshold, issue_code
- 现象：两个相邻信控路口距离较短，排队容易互相传导。
- 判别依据：相邻路口间距较短、上游排队接近下游出口、绿波通过率偏低。
- 适用边界：适用于干线或密集路网中的连续信控路口。
- 不要误判为：单点周期不足；若下游出口被违停或施工占用，应同时判断秩序或事件干扰。
- 建议生成：diagnosis_checklist, thresholds, rules_md, script_draft
- 运行时意图：触发短间距协调约束检查，必要时提升 green_wave_break 或 downstream_blockage 优先级。
```

## 手动编译命令

默认只生成产物和 diff，不写回运行时文件：

```bash
python tools/expert_compiler/compile_expert_experience.py \
  --source skillpacks/intersection/common/路口专家经验规则.md \
  --target skillpacks/intersection
```

生成内容位于 `.generated/expert_compiler/`：

- `expert_rules.ir.json`：结构化中间表示。
- `expert_compiler.diff`：建议写回运行时文件的 diff。
- `skillpacks/intersection/...`：渲染后的文件快照。

确认 diff 后写回：

```bash
python tools/expert_compiler/compile_expert_experience.py \
  --source skillpacks/intersection/common/路口专家经验规则.md \
  --target skillpacks/intersection \
  --apply
```

检查源稿和生成产物是否同步：

```bash
python tools/expert_compiler/compile_expert_experience.py \
  --source skillpacks/intersection/common/路口专家经验规则.md \
  --target skillpacks/intersection \
  --check
```

## 生成产物说明

| 产物 | 用途 | 写回策略 |
| --- | --- | --- |
| `references/rules.md` | 生成专家经验自动区块，供 LLM 与工程规则读取 | `--apply` 可写回 |
| `diagnosis_checklist.yaml` | 新增检查项草案 | `--apply` 可写回，带 `requires_human_review` |
| `scene_cognition_checklist.yaml` | 新增场景认知检查项草案 | `--apply` 可写回，带 `requires_human_review` |
| `expert_rule_source_map.yaml` | 记录专家条目和生成目标映射 | `--apply` 可写回 |
| `SKILL.md` | 生成执行步骤或引用文件调整提示 | `--apply` 可写回提示区，不直接改 frontmatter |
| `scripts/generated/*.py` | Python 检查逻辑草稿 | 只作为草稿，人工审核后迁入正式脚本 |

## 审核要点

- `source_id` 是否稳定且唯一。
- 新增检查项是否有合理的 `profile_field`、`issue_codes` 和 `cause_dimensions`。
- 阈值是否已存在于 `thresholds.yaml`，新增阈值是否有单位和阶段语义。
- 自动生成的 `rules.md` 文案是否符合专家意图。
- Python 草稿是否只在 `scripts/generated/` 中，不能直接被运行时引用。

## 不建议专家直接修改

- `SKILL.md` 的 frontmatter、`execution_steps`、脚本路径和函数名。
- Python 脚本中的字段解析、SQL 查询和输出结构。
- `references/data_dictionary.md` 中的 PG 表和字段映射。
- 前端或 API 的运行时调用路径。
