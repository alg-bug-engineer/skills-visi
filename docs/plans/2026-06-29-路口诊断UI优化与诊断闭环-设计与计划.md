## 1. 设计原则

- **目标用户**：信控/交警领导演示 + 工程师复核。深色作战台风格，已有设计语言沿用，不做推倒式改版；用 frontend-design 原则提升**地图动作可读性**与**关键时刻动效**。
- **诊断驱动呈现**：所有卡片/图层/播报都从「当前诊断出的问题类型(`problem_types`) + 激活维度包」派生——**用不到的维度不出现**。这是 #2/#3/#7 的统一主线。
- **播报=分句流，不截断**：长文按句切成多条 cue 顺序播，面板展示全文，语音读全要点。
- **量化权威在规则**：LLM 只润色与对齐，秒数来自 `action_plan`，删除硬编码公式回退。

## 2. 分阶段方案

### 阶段 P1 · 语音播报修复（诉求 #1、#2 的播报部分）

**问题**：`summarizeNarrationForVoice` 把每条旁白压到 42 字并 `.slice()`，长「路口结构」被切；timing 分支强行播「周期X秒（即配时206s）」。

**方案**：
- 新增 `splitNarrationToCues(phase, text, title)`：按 `。；\n` 分句，过滤列表符号，**逐句成 cue**（保留语义完整），替代单条硬截断。每句仍可设软上限（如 60 字）但**按标点断句**而非粗暴切字符。
- timing 分支**移除** `周期X秒` 拼接；timing 播报只读「配时方案要点」中**与当前问题相关**的部分（空放才读绿信比/空放；拥堵读供需匹配，不读周期秒数）。
- `useVoiceNarration` 已支持 cue 队列与严格栅栏，分句 cue 直接入队即可；`stepIndex/phase` 去重逻辑保持。

**校验**：`voiceTextSummarize.spec.ts` 扩用例（长结构文案不截断、timing 不含「周期…秒」、空放才读绿信比）；手动跑一轮听感。

---

### 阶段 P2 · 诊断驱动的卡片/图层裁剪（诉求 #2 卡片部分 + #7 地图动作底座）

**核心**：引入**前端「激活维度」单一事实源**，由后端 `problem_types` + 维度包推导，驱动卡片与地图图层显隐。

- 后端：`/diagnose` 响应（或 SSE meta）补 `active_dimensions: string[]` 与 `problem_types: string[]`（dimension_pack_service 已有数据，仅需 surface）。
- 前端：新增 `useActiveDimensions()` composable，集中判断「某卡片/图层是否在当前问题下相关」。
  - insight 卡（TimingProfileCard / FlowTimingGovernanceCard / ConstraintStackCard 等）与渠化底部浮层（图例、运行数据卡、环图）统一接入 `isDimensionActive(dim)`，**无关即不挂载**（非 `v-show`，真正不渲染，省篇幅）。
  - 底部卡片从左到右按「激活维度优先级」排序；空放才出配时方案+环图，拥堵不强行带环图。

**校验**：四类问题各跑一次，断言卡片集合差异（拥堵无环图、空放有环图+绿信比、溢出有排队/下游、冲突有相位相序）。

---

### 阶段 P3 · 渠化流量与排队呈现（诉求 #3，POI/AOI 除外）

- 渠化图（`channelization/*` + `MapStage`）叠加两类**地图动作图层**：
  - **流量层 flow**：进口道按转向流量/饱和度着色 + 数值标注（数据已在 evidence/traffic_flow）。
  - **排队层 queue**：用 `buildQueueDataFromEvidence` 产出按进口道画排队长度条（米/车数），高饱和进口强调。
- 统一进 P4 的地图动作调度器（见下），P3 先把数据通路与图元画出来。

**校验**：渠化态下流量数值与排队条与后端 evidence 对齐；无数据进口降级为灰显不报错。

---

### 阶段 P4 · 统一地图动作层（诉求 #7）

- 新增 `mapActionLayer`（或扩 `mapHighlight.ts`/`mapMarkers.ts`）暴露统一动作词汇：
  `mark(point,label)` 标注 / `emphasize(target)` 强调 / `queue(arm,len)` 排队 / `flow(arm,val)` 流量 / `clear(scope)`。
- 与播报严格栅栏对齐：每步播报点亮对应地图动作（识别→mark 问题进口；归因→emphasize 成因；方案→flow/queue 前后对比）。
- frontend-design 原则落点：动作进出用 staggered reveal + 描边/脉冲强调，关键时刻（出方案）一次编排动效，而非零散微交互。

**校验**：四类问题各自地图动作脚本走查；动作与 voice cue 步骤一一对应。

---

### 阶段 P5 · 经验库三 tab（诉求 #5）

- 后端：新增 `GET /experience/library?inter_id=...`（或复用既有 session/profile 接口）聚合 `IntersectionProfileStore` 的三桶：
  `cognition[]`（认知经验）、`diagnosis[]`（诊断经验）、`solution_ref[]`（方案经验，关联 skill）。
- 前端：`SkillLeaderboardPanel` → 重命名/重构为 `ExperienceLibraryPanel`，标题「经验库」，下设三子 tab：
  - **认知经验**：问题记录（verified/data_doubt/manual 态徽标）。
  - **诊断经验**：成因先验（dimension/scope/confidence）。
  - **方案经验**：量化方案（qualitative↔quantified，跳转 skill）。
- 复用高亮沿用既有 `reused_experience` 机制：tab 内对「本轮被复用」的条目点亮。

**校验**：三 tab 数据来自真实 profile；空态友好；复用高亮与本轮诊断一致。

---

### 阶段 P6 · 建议生成闭环（诉求 #4）

- 删除/弃用 `evaluate_formula` 的 `min(sat*12,25)` 作为**展示量化**的来源：`SuggestionService.generate` 的 `delta` 必须来自 `action_plan`（governance 推导），formula 仅作无 action_plan 时的兜底且不进面板措施行。
- 核对 orchestrator：确认 `case_experience`（来自 `case_library_service` + `expert_knowledge.md`）与 `delta_override`/`direction_override` 确实传入 `generate`；补缺。
- 建议正文随 `problem_types` 动态：拥堵/空放/溢出/冲突走不同 guidance 块（governance_guidance 已分维度），确保「旧版本文案」被替换。

**校验**：四类问题建议正文差异化；措施行秒数=action_plan；案例库经验在 meta.case_experience 出现且影响表述。

---

### 阶段 P7 · 四类问题端到端验证与沉淀补强（诉求 #6）

- 逐类型走查：**拥堵 / 空放 / 溢出 / 冲突** 各构造一条用户输入，断言：
  1. NLU 命中正确 `problem_types`；2. 激活维度包正确；3. 规则诊断有结论（修复未打 `focus_category` 的拥堵规则 green_insufficient/approach_high_delay/cycle/plan_granularity）；4. 三级经验（cognition/diagnosis/solution_ref）均落库；5. 严格栅栏播报与之对齐。
- 补 `focus_category` 标签；补端到端测试（每类型一条 e2e）。

**校验**：四类各一条 e2e 全绿；profile JSON 三桶均有新增；后端既有 204 测试不回归。

## 3. 实施顺序与依赖

```
P1(语音, 独立) ─┐
P2(维度裁剪) ──┼─> P3(流量/排队) ─> P4(地图动作层)
P6(建议闭环) ──┘                    P5(经验库三tab, 依赖后端聚合接口)
P7(四类验证) 贯穿，作为各阶段的验收闸
```

建议落地批次：**B1 = P1+P6**（播报+建议，纯逻辑、风险低、演示最直观）→ **B2 = P2+P3+P4**（诊断驱动呈现+地图）→ **B3 = P5+P7**（经验库+四类验证收口）。

每批次走：实现 → 后端 pytest / 前端 vitest 绿 → `npm run build`(vue-tsc，注意既有类型错误基线) → 手动演示一轮。

## 4. 风险

- 前端 `npm run build` 既有类型错误基线（spec 夹具/未用变量），改动需不新增错误，不强求清零既有。
- 卡片「真不渲染」可能触发既有布局/动效假设，需回归 narrativeStack/footerLayout 用例。
- `focus_category` 补标可能改变既有诊断触发面，需与 204 既有测试对照防回归。
- 地图动作层与绿波 CorridorWave 等既有图层共存，避免图层 z-index/清理冲突。

## 5. 已确认（2026-06-29 需求方）

- 批次顺序：**B1(P1+P6) → B2(P2+P3+P4) → B3(P5+P7)**。
- 经验库聚合接口：**新开 `GET /experience/library`**（按 inter_id 返回三桶）。
