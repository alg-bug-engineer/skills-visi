# 信控叙事主轴重构 · 供需匹配度为主轴

> 日期：2026-06-29 · 状态：**已实现**（见 [复盘](./2026-06-29-信控叙事主轴重构-供需匹配度-复盘.md)）
> 关联：`docs/PROJECT_OVERVIEW.md` §3.1「四类信控问题」、`docs/路口专家经验规则.md`、`docs/路口指标证据计算说明.md`
> 决策记录：叙事主轴 = 方案A（供需匹配度）；全饱和 = 诚实标注瓶颈 + 给非配时出路；交付 = 文案/空数据/叙事三块全做，叙事为重。

---

## 1. 背景与问题

当前四类信控问题（饱和度 / 失衡 / 空放 / 溢出）以**并列罗列**方式叙事：
`FlowTimingGovernanceService._build_summary` / `_governance_narrative` 先列"命中了哪几类"，
饱和度作为其中一类与其他并列。

**缺陷**：叙事以「饱和度高低」（总量）为落点。但信号配时是**绿灯时间的再分配工具，不创造通行能力**。
于是出现一个无法自洽的场景——

> 四个进口流量都大、都饱和时，"饱和度命中"这句话成立，但**调配时帮不上忙**，叙事落不到信控动作上。

根因：**总量饱和度不是信控的可操作量；绿灯分配是否错配才是。**

## 2. 核心洞察

Webster 配时理论的本质：**最优配时让各关键转向的饱和度趋于相等**。由此：

- **饱和度"分布不均"（spread 大）** = 绿灯错配 = **配时可解决**（把绿灯从空放方挪给过饱和方）。
- **饱和度"分布均匀且普遍高"（spread 小、max 高）** = 真实能力瓶颈 = **配时近最优、单点无解**，出路在加周期/绿波/渠化/需求。

因此叙事主轴从「饱和度高不高」改为「**这是不是配时能解决的问题、该往哪个转向挪绿灯**」。
饱和度退居 **severity（严重度）背景**，不再做头牌。

## 3. 数据支撑（可行性已核实）

粒度 = **转向级（`dir8_code` × `turn_dir_no`）× 5 分钟**，供需两侧同粒度齐备：

| 维度 | 表 | 字段 | 角色 |
|------|----|------|------|
| 需求 | `dws_inter_link_turn_flow_5min_mm` | `turn_flow_total` | 各转向流量 q_i |
| 需求 | `dws_turn_saturation_5min_mm` | `turn_saturation` | **各转向饱和度 x_i**（max/min/spread 已算） |
| 供给 | `dws_turn_min_green_5min_mm` | `green_time_plan` / `min_green_time` | 各转向绿灯 / 最小绿 |
| 供给 | `dwd_ctl_inter_plan_cfg` / `_stage_timing` | `cycle_len_sec` / 相位 | 周期 C、相位结构 |
| 反证 | `dws_turn_green_utilization_5min_mm` | 绿灯利用率 | 空放（可重分配的"供绿方"） |
| 出路 | `dws_corridor_coord_*` | 协调配置 | 绿波（单点失效时的下一步） |

**已就绪指标**（`data_fetcher.py` 输出 `traffic_flow.*`）：
`turn_saturation_max`、`turn_saturation_spread`、`saturation_rate`、`lane_saturation_max`。
**结论**：主轴所需指标零新增数据，全部已落在 payload 中。

**已知 caveat（不阻塞）**：
- `dws_lane_capacity_5min_mm`（饱和流率 s_i 代理）偶为空，已标 `lane_capacity_missing` data-gap。
  spread 分析不依赖单独 capacity——`turn_saturation` 已把 s_i 吸收进去。
- `flow_green_check` 当前用 Spearman 秩相关（弱口径），可后续升级为 λ_i（绿信比）vs y_i（流量比）直接比；
  本轮不依赖该升级，仅作佐证。

## 4. 叙事主轴重构逻辑

`FlowTimingGovernanceService.build()` 新增 **主诊断 `primary_diagnosis`**，作为叙事头牌，四选一：

| type | 判定（复用现有阈值） | severity | 信控含义 / 落点 |
|------|------|---------|----------------|
| `timing_optimizable` 配时可优化·绿信比错配 | `spread ≥ 0.30` 且 `max_x ≥ 0.80` | high if `max_x ≥ 0.90` else medium | 高饱和转向与空放/低饱和转向并存 → **把绿灯从空放方挪给过饱和方** ← 信控接单 |
| `capacity_bottleneck` 能力瓶颈·配时近最优 | `spread < 0.30` 且 `max_x ≥ 0.90` | high | 各转向饱和度接近且普遍高 → 绿信比近最优、**单点配时空间有限** → 出路：加周期 / 绿波协调 / 渠化展宽 / 需求调控 |
| `structure_limited` 配时结构受限 | 上述任一 **且** 关键转向触最小绿（`green_deficit_ratio_max > 0`） | 叠加标注 | 关键转向绿灯已触最小绿，缩无可缩——叠加在主诊断上，不单独成型 |
| `basically_matched` 基本匹配 | `max_x < 0.80` | none | 供需与配时基本匹配，维持现状持续监测 |

**输入量**（均已在 payload）：
- `max_x` = `traffic_flow.turn_saturation_max`（缺失回退 `saturation_rate` / `lane_saturation_max`）
- `spread` = `traffic_flow.turn_saturation_spread`
- 佐证：`timing_profile.flow_green_fit.verdict`（weak/mismatch 印证错配）、
  `evaluation.green_utilization` / 空放转向（指认"供绿方"）、`timing_profile.green_deficit_ratio_max`

**阈值来源**（`rules/thresholds*.yaml`，统一经 `thresholds_loader`）：

| 语义 | key | 现值 | 处理 |
|------|-----|------|------|
| 饱和度高 | `saturation.high` | 0.80 | 复用 |
| 过饱和 | `saturation.oversaturation` | 0.90 | 复用 |
| 失衡/分布不均分界 | `imbalance.diagnosis` | 0.30 | 复用为 spread 分界 |
| 最小绿亏空 | `timing.green_deficit_ratio` | 0.10 | 复用 |

> ⚠️ spread 0.30 / max 0.90 两个分界需用**演示主秀路口（会展路与奥体中路，`011wwe29jbf00001`，锚定 2026-06-13 晚高峰）**真实数据校验后定稿；
> 若分界对实际数据不合理，仅调 `thresholds_extensions.yaml`，不改判定结构。

## 5. 输出契约变更

### 5.1 后端 `FlowTimingGovernanceService.build()` 返回新增

```python
"primary_diagnosis": {
    "type": "timing_optimizable" | "capacity_bottleneck" | "structure_limited" | "basically_matched",
    "headline": str,        # 头牌结论（去术语，见 §6）
    "lever": str,           # 信控动作落点；basically_matched/capacity_bottleneck 为非配时出路
```

`summary` / `governance_narrative` 改为**以 `primary_diagnosis.headline` 开头**，四类 problems 退为补充扫描。

## 6. 文案模板（含去术语）

> 全程**禁止暴露数据仓术语**（DWS / DWD / 表名 / step_index / dir8 等）。面向交警，用业务语言。

| type | headline 模板 | lever 模板 |
|------|--------------|-----------|
| `timing_optimizable` | 「{过饱和转向}已过饱和（{x:.2f}），而{空放转向}绿灯有富余——属于绿灯分配不均，配时可改善」 | 「建议把{空放转向}的部分绿灯时间让给{过饱和转向}，优先调整绿信比分配」 |
| `capacity_bottleneck` | 「各转向饱和度接近且普遍偏高（最高 {x:.2f}），该路口已接近通行能力上限」 | 「绿信比分配已接近最优，单点配时优化空间有限；建议从加大周期 / 绿波协调 / 车道展宽 / 需求调控入手」 |
| `structure_limited`（叠加） | 在头牌后追加「；其中{转向}绿灯已触最小绿，压缩空间有限」 | — |
| `basically_matched` | 「供需与配时基本匹配，未见明显绿灯错配」 | 「维持现有配时方案，持续监测高峰表现」 |

## 7. 空数据隐藏规则

**原则**：检查单维度（四类 chip）可显示"正常"；**具体证据行/字段**无数据则不显示。

| 对象 | 规则 |
|------|------|
| 四类 chip（`GovernanceDimChips`） | 保留命中/正常（检查单扫描语义） |
| `problem.evidence[]` 行 | 已有 `_evidence_lines` 过滤空串——核对所有 `_line(cond, text)` 的 cond 严格对应数据存在 |
| `primary_diagnosis.evidence[]` | 仅 push 有值项；`max_x` / `spread` 缺失时不输出该证据行 |
| `flow_green_tau` / `match_narrative` | 值为 None / `insufficient` 时前端不渲染该行，不显示"暂无法评价" |
| `deficit_turns` / `expert_rules` | 空数组时整块不渲染（前端 `v-if="length"`） |
| `data_gaps` | 不直接展示原始 key（如 `lane_capacity_missing`），如需提示则映射为业务语言 |

## 10. 风险与边界

- **阈值经验性**：spread 0.30 / max 0.90 为初值，须演示数据校验；改阈值不改结构（§4 末注）。
- **s_i 缺失**：lane_capacity 偶空，主轴不依赖；λvs y 精确口径留作后续增强，非本轮范围。
- **不在本轮**：`flow_green_check` 升级为绿信比/流量比直接比；区域协调（绿波）联动诊断的自动化推荐——本轮仅在 `capacity_bottleneck` 文案中作为"出路"指向，不做联动计算。
- **回归**：四类 chip 扫描行语义不变，避免破坏既有演示节奏与 `voice_narration` 步骤对齐。
