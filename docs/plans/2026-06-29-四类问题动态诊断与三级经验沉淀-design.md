# 四类问题动态诊断与三级经验沉淀 · 重构设计

日期：2026-06-29
分支：main（标签 `0630`）
状态：已实施并发布 @ `0630`

---

## 1. 背景与目标

当前演示系统叙事「头重脚轻」：前段（NLU、场景认知、证据铺陈、地图）很重且同质，
末段「经验吸收」只是末端一次性的 skill 固化与终端流水展示，没有真正的沉淀增值。

本次重构目标：

1. **NLU 四类问题分类 + 维度动态裁剪**：用户问题分为「拥堵 / 溢出 / 空放 / 冲突」四类；
   流量 / 渠化 / POI / AOI 为所有问题共有的基础底座，再按问题类型挂载维度子集。
2. **三级经验的「吸收—量化—复用」闭环**：认知经验、诊断经验、方案经验在理解过程中
   **逐级解锁、持续沉淀**，复用时分类型按步注入并显式高亮。
3. **叙事重心再平衡**：压缩前段同质证据，把篇幅让给「问题→诊断→方案」主线。
4. **删除干线扫描（corridor scan）**：该入口不再需要。
5. **严格栅栏播报**：播报与理解步骤一一绑定，每步产出落库后才播报。

### 决策记录（需求方确认）

| 议题 | 决策 |
|---|---|
| 认知/诊断经验载体 | **路口认知档案 JSON 快照**（方案经验沿用 skill 固化） |
| LLM 职责边界 | **脚本/规则判定为准**，LLM 做分类 + 经验抽取归一化 + 叙事润色 |
| 重构方式 | 现有 backend **增量重构**（保留 orchestrator 骨架） |
| 播报同步 | **严格栅栏**：每步完成才播报对应结论 |
| 多类型并存 | **允许维度包叠加**（base + 多包，如「又堵又溢」） |
| knowledge_qa 接入 | **先轻量匹配**（关键词 / 路口形态），RAG 留后续 |

## 2. 现状关键事实（已核实）

运行时主链路（`backend/intersection_agent/`，非 `docs/intersection` 技能源码）：

```
handle_message → _run_nlu → _run_pipeline(
    DataFetcher → cognition → timing_profile → evidence → sustained →
    RuleEngine.diagnose_comprehensive → FlowTimingGovernance
) → _diagnose → _generate_suggestion_content → skill 固化
```

- `nlu_service.py:33` 提示词写死「只做拥堵诊断」；`models/domain.py:58` `NluResult.problem_type`
  与 `domain.py:50` 默认 `"congestion"` 已存在，可直接扩展为四类。
- `RuleEngine.diagnose_comprehensive()` 跑全量检查，无按问题类型裁剪。
- LLM 已接入（`llm/qwen_client.py`），多服务已使用（governance/nlu/follow_up 等），无需新建客户端。
- `skill_service.py` 仅末端把 session 打包成 skill，无三级经验概念。
- `stores/` 仅有内存级 `SessionStore`，缺少路口级持久档案。
- deepagent 的检查单驱动诊断（`skillpacks/intersection/common/diagnosis_checklist.yaml`、
  `cause_dimension_map.yaml`、`thresholds.yaml`、`scene_type_rules.yaml`）与四类问题高度对应，可移植。

## 3. 目标架构

### 3.1 新链路

```
NLU(LLM 分类四类问题, 允许多类型)
  → 加载基础维度 base(流量/渠化/POI/AOI)
  → 注入该路口【认知经验】(profile 档案)
  → 激活问题类型维度包(可叠加)
  → 规则脚本诊断 + 六维成因评分
  → 注入【诊断经验】先验
  → 方案生成(规则量化 + knowledge_qa 轻量匹配)
  → 沉淀【方案经验】(skill) + 档案记 solution_ref
  → 全程严格栅栏播报
```

### 3.2 维度子集配置层（移植 deepagent）

新增 `backend/intersection_agent/config/problem_dimension_packs.yaml`：

- `base`（恒在）：流量、渠化、POI、AOI、相邻间距。
- `congestion`（拥堵）：饱和度、绿灯利用率、排队长度、延误指数、需求压力。
- `spillback`（溢出）：排队长度、溢出指标（queue_storage_ratio）、下游阻塞。
- `empty_green`（空放）：绿灯利用率、配时方案、环图、绿信比失衡、周期。
- `conflict`（冲突）：相位相序、渠化匹配（phase_channel_mismatch）、机非冲突。

移植来源：deepagent `diagnosis_checklist.yaml`(item→issue_codes→cause_dimensions)、
`cause_dimension_map.yaml`(issue→六维)、`thresholds.yaml`(阈值)。

`RuleEngine`：新增 `diagnose_by_dimensions(problem_types: list[str])`，按 base + 激活包
跑命中项；`diagnose_comprehensive` 保留为内部全量回退。

### 3.3 三级经验层（核心新增）

新增 `stores/intersection_profile_store.py`，文件级 JSON，每路口一档：

```jsonc
{
  "inter_id": "xxx",
  "cognition": [        // 认知经验：问题记录
    { "text": "下午四点常堵", "status": "verified|data_doubt|manual",
      "evidence": {...}, "source": "user|data", "ts": "..." }
  ],
  "diagnosis": [        // 诊断经验：原因解释先验
    { "cause": "旁边小学放学", "dimension": "event", "scope": "...",
      "source": "user", "confidence": 0.x, "ts": "..." }
  ],
  "solution_ref": [     // 方案经验指针（实体固化在 skill 包）
    { "skill_id": "...", "qualitative": "绿灯多给点",
      "quantified": "东进口绿灯 +8s", "ts": "..." }
  ]
}
```

写入时机（逐级解锁）：
- 识别问题步 → 写 `cognition`（数据可验证=verified；不可验证但人坚持=data_doubt）。
- 归因步 → 写 `diagnosis`。
- 出方案步 → 量化结果固化 skill，并在档案追加 `solution_ref`。

复用时机（分类型按步注入）：
- 识别问题步注入 `cognition`；归因步注入 `diagnosis`；
  出方案步注入 `solution_ref` + `knowledge_qa` 轻量匹配范例。
- UI 显式高亮「本步复用了 XX 路口 / 历史的某条经验」。

LLM 职责：把「绿灯多给点」等定性反馈**抽取归一化**为结构化条目；
把规则算出的 `+x 秒` 与定性描述对齐润色。判定权威仍在规则脚本。

### 3.4 knowledge_qa 轻量匹配

`docs/knowledge_qa.jsonl`（1314 条：案例场景/交通问题诊断/治理方案/预期效果）。
新增 `services/case_library_service.py`：按问题类型关键词 + 路口形态（短间距/溢出/相位等）
做关键词与标签匹配，返回 Top-K 范例供方案生成参考。**不引入向量库依赖**。

### 3.5 严格栅栏播报

播报状态机与理解步骤一一绑定：`emit(step_conclusion)` 仅在该步骤产出**落库后**触发。
前端 `ProcessTimeline` / `ExperienceAbsorptionPanel` 改为「沉淀型」——
每步把新沉淀的经验条目实时点亮，而非纯文字流。

### 3.6 删除干线扫描

后端移除：`corridor_scan_service` / `corridor_scan_nlu_service` / `corridor_scan_scene` /
`corridor_context_service` / `corridor_narrative_service` / `corridor_pick_resolver` /
`line_resolver`（干线部分）。orchestrator 摘除 `_run_corridor_pipeline` /
`_handle_corridor_pick` / `_continue_corridor_scan`。
前端移除 `components/corridor/` 与相关入口。（保留 `region-scan` 独立项目，仅去干线入口。）

### 3.7 叙事再平衡

`problem_evidence_service` 仅保留与激活维度相关的证据；省下的篇幅让给后段
「问题→诊断→方案的吸收—量化」主线。

## 4. 任务拆解

**阶段 A · 维度化诊断**
- T1 `problem_dimension_packs.yaml`：四包 + base，移植 deepagent 检查单/阈值/成因映射。
- T2 NLU 改造：提示词支持四类分类、允许多类型；`problem_type` → `problem_types: list`。
- T3 `RuleEngine.diagnose_by_dimensions()`；pipeline 接入；证据按激活维度裁剪。

**阶段 B · 三级经验层**
- T4 `IntersectionProfileStore`（JSON 档案 + schema + 读写）。
- T5 三级写入挂点（识别/归因/出方案步），含 status 判定逻辑。
- T6 分类型复用注入 + UI 高亮「经验复用」。
- T7 LLM 经验抽取归一化 prompt（定性→结构化，定性↔量化对齐）。

**阶段 C · 案例库与方案量化**
- T8 `case_library_service` 轻量匹配；方案生成接入 Top-K 范例。

**阶段 D · 播报与前端**
- T9 严格栅栏播报状态机；emit 绑定步骤落库。
- T10 `ProcessTimeline`/`ExperienceAbsorptionPanel` 改沉淀型展示。
- T11 删除干线扫描（后端服务 + orchestrator 挂点 + 前端组件）。

**阶段 E · 收尾**
- T12 叙事再平衡（前段证据压缩）；端到端演示走查；阈值校准。

---

## 5. 风险与待校准

- 维度包阈值（饱和度/溢出/绿灯利用率）需用演示数据校准（参考 deepagent thresholds）。
- 多类型叠加时的优先级与去重，避免重复证据。
- 认知经验 `data_doubt` 态的呈现口径（避免与已证实问题混淆）。
- 关键词匹配召回质量；必要时为 knowledge_qa 预生成标签。
