## 1. 背景与目标

诊断到「某进口道过饱和但本地已无信控可调空间」时，结论往往停在口号式的"需上游协调"。
本功能把它落到具体路口：**输入路口 + 进口道方向，自动溯源上游、逐级体检、定位可信控治理落点**，
并以"地图运镜 + 旁白分步叙事"的产品语言演出整条推理链。

目标：

1. **进口道级指标**：计算指定进口道的转向饱和度、绿灯利用率、排队长度等。
2. **过饱和触发溯源**：饱和度 ≥ 阈值时，沿干线找上一跳上游路口及其左/直/右流量与占比。
3. **可治理判定 + 递归上溯**：体检上游路口四向饱和度/排队，判断是否有信控优化空间；
   若一跳上游四向全饱和（无空间），递归分析其**另外三个进口道**各自的直接上游（第二跳），**最多二跳**。
4. **作为诊断对话流的一个 phase**：本路口有问题或某进口道过饱和则自动触发；多方向过饱和则并行分析、串讲呈现。
5. **强交互可视化**：地图缩放到道路级、沿干线高亮加粗、节点四向饱和环、可暂停/回放/钻取。

### 1.1 决策记录（需求方确认）

| 议题 | 决策 |
|---|---|
| 形态 | **诊断对话流的一个 phase**（非独立页面），插在 `diagnose` 之后、`confirm_bubble` 之前 |
| 触发 | **自动**：本路口问题成立 或 任一进口道 `turn_sat ≥ trigger_saturation` |
| 多方向 | **并行分析**，呈现用 **逐树串讲**（先讲完一棵树再讲下一棵） |
| 回放语义 | **帧重建**：地图状态 = f(帧号)，拖回则撤销后续覆盖物，可重现 |
| governable 定义 | **非四向全饱和**（有进口道 `turn_sat < full_saturation` 即有绿信比可借调），叠加 `green_util < 0.5` 作为更强落点信号 |
| 递归深度 | **最多二跳**（`max_hops = 2`），二跳无条件截止 |
| 实现方式 | **复用增量扩展**：复用 `data_fetcher`/`flow_trace_service`/`MapStage`，不重写数据访问与地图 |

## 2. 核心逻辑：向上游搜索可治理落点的递归树

把每个路口节点抽象为：

```
node = {
  inter_id, inter_name, feeding_dir8,        # 该节点经哪个进口道喂给下游
  approach_profiles: [4个进口道 {dir8, turn_saturation_max, queue_len_est_m, green_util}],
  upstream_flow: {left_pct, through_pct, right_pct},  # 喂给下游的左/直/右占比（缺失降级）
  saturated_all: bool,                        # 四个方向是否全饱和
  governable: bool,                           # 是否有信控优化空间
  decision: "治理落点" | "继续上溯" | "二跳截止",
  hop: int,
  children: [node...]                         # 仅当 !governable 且 hop < max_hops
}
```

递归算法（终止条件）：

```python
def analyze(inter_id, feeding_dir8, hop, window) -> Node:
    profiles = approach_profiles(inter_id, window)          # 复用 data_fetcher（4 个进口道）
    saturated_all = all(p.turn_sat_max >= FULL_SAT for p in profiles.four_dirs)
    governable = (not saturated_all) or any(p.green_util < GOV_GREEN_UTIL for p in profiles)
    node = build_node(inter_id, feeding_dir8, profiles, hop, governable, saturated_all)

    if governable:
        node.decision = "治理落点"                          # 此处可借调绿信比
    elif hop >= MAX_HOPS:
        node.decision = "二跳截止"
    else:
        node.decision = "继续上溯"
        for ap in other_three_approaches(inter_id, exclude=feeding_dir8):
            up = upstream_one_hop(inter_id, ap, window)     # 复用 lock_one_hop（同方位 coverage 最大）
            if up:
                node.children.append(analyze(up.inter_id, up.feeding_dir8, hop + 1, window))
    return node
```

**入口语义对齐需求**：
- `hop0` = 目标路口 + 输入进口道，**假设无信控空间**，只算指标并作触发器（`turn_sat ≥ trigger_saturation` 启动溯源）。
- `hop1` = 目标进口道直接上游 → 判 governable。
- `hop2` = 仅当 hop1 四向全饱和时，对其**另外三个进口道**各找直接上游。
- 硬截止于 `hop2`。

**多方向并行**：诊断阶段收集所有过饱和进口道 `[北进口, 东进口, …]`，对每个并发 `asyncio.gather(analyze(...))`，得到 N 棵树。

## 3. Storyboard 帧模型（逐树串讲 + 帧重建的共同底座）

"帧重建"要求每帧状态可由帧号确定地算出；"逐树串讲"决定帧排列顺序。两者合起来：
**后端一次性产出完整 storyboard，前端本地播放**（不逐帧 SSE 推送，否则拖回时早帧未必已到，无法重建）。

```jsonc
storyboard = {
  trees: [
    { tree_id: "N", approach: "北进口",
      nodes: [ {id, inter_id, name, lon, lat, role, approach_profiles[], decision} ],
      edges: [ {id, from, to, path:[[lon,lat]...], flow_pct, dominant_turn} ] },
    { tree_id: "E", approach: "东进口", ... }
  ],
  frames: [                         // 扁平有序，逐树串讲
    {idx:0, tree:"N", kind:"thesis",   focus:"target", reveal:[],                       camera:"intersection", narration:"北进口直行饱和0.95、排队142m、绿用0.38——本地无可调空间，向上游溯源。"},
    {idx:1, tree:"N", kind:"zoom_out", focus:"target", reveal:[],                       camera:"corridor"},
    {idx:2, tree:"N", kind:"hop1",     focus:"up1",    reveal:["edge:N-up1"],           camera:"corridor", narration:"约67辆/100来自上游XX路口，以直行为主。"},
    {idx:3, tree:"N", kind:"inspect",  focus:"up1",    reveal:["ring:up1"],             narration:"该路口四向全饱和，继续上溯。"},
    {idx:4, tree:"N", kind:"hop2",     focus:"up1",    reveal:["edge:up1-A","edge:up1-B","edge:up1-C","ring:A","ring:B","ring:C"]},
    {idx:5, tree:"N", kind:"resolve",  focus:["A","B"],reveal:["badge:A★","badge:B★"],  camera:"fit_tree_N", narration:"定位2个治理落点：A路口西进口、B路口南进口可借调绿信比。"},
    {idx:6, tree:"E", kind:"thesis",   ...}            // 切到东进口树：先 establishing 广角再收
  ]
}
```

**核心约定**：
- `reveal` 是**累计可见集**。`renderToFrame(storyboard, n)` = 绘制 `frames[0..n]` 内所有 `reveal` 覆盖物的并集，仅高亮当前 `frame.tree`、其余树淡化（`dimOthers`）。
- 渲染是纯函数、幂等、无副作用 → 帧重建天然成立。
- 控制条的播放/暂停/单步/拖拽**全部只改 `n`**。
- 切树（如 idx 5→6）：上一棵树淡出，相机给一个 establishing 广角，再收到新树。

## 4. 控制条状态机（回放语义）

```
状态: { n: 当前帧, playing: bool, speed: 1x|2x, userEngaged: bool, showHop2: bool }

自动播放: playing=true 时定时器 every (BASE_MS / speed) 执行 n = min(n+1, last)，到尾自动 playing=false
单步:    ▶▶ → n=min(n+1,last)；◀◀ → n=max(n-1,0)；均隐含 playing=false
拖拽:    进度刮擦直接 set n（帧重建重绘），隐含 playing=false
让位规则: 用户滚轮/拖拽地图 → userEngaged=true，暂停自动运镜与相机 setFitView，
         避免与现有 MapStage "用户介入则不强制对位" 行为冲突（见 MapStage 现有注释）
展开二跳: showHop2 切换 → 影响 hop2 相关 reveal 是否纳入 renderToFrame（减负）
```

控制条 UI：`◀◀  ⏯  ▶▶   ●━━━○──────  3/6   [展开二跳 ⃝]  [1x▾]`。
复用/扩展现有 `demo_pacing` 节奏控件，不另造播放器。

### 7.3 阈值配置（新增）

```yaml
upstream_trace:
  trigger_saturation: 0.90      # 目标进口道触发溯源（与 flow_trace 对齐）
  full_saturation: 0.85         # 单进口道"算饱和"下限
```

### 7.4 容错

- 溯源/任一查询失败 **仅告警不阻断主诊断**（沿用 `flow_trace_service` 现有 try/except 容错）。
- `flow_share_ratio` 缺失 → 占比降级为空、边按默认权重（沿用 `_as_float` 容错）。
- `MOCK_DB` 路径：构造"一跳全饱和 + 二跳含治理落点"的确定性 mock，便于无库联调与单测。

## 9. 测试要点

**后端**
- 递归终止：governable 即停 / 全饱和则上溯 / hop2 截止。
- governable 判定边界：四向全饱和 vs 含未饱和 vs 含 green_util 空槽。
- 多进口道并行：N 棵树独立、互不串扰。
- storyboard：`reveal` 累计正确、帧顺序为逐树串讲、边 path 取自 link。
- 容错：溯源失败 / flow_share 缺失 / MOCK_DB 确定性。

**前端**
- `renderToFrame` 幂等：相同 n 多次渲染结果一致；拖回撤销后续覆盖物。
- 逐树串讲：一次仅一棵树高亮；Tab 切换跳首帧。
- 让位：用户介入地图 → 暂停自动运镜与相机对位。
