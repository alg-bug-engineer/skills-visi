# 四类信控问题诊断与阿里云 ECS 部署 — 需求说明

> 版本：2026-06-26  
> 状态：已定稿，进入开发

---

## 1. 背景

路口问题诊断智能体已具备规则引擎、证据计算、配时画像与 Skill 固化能力。为进一步支撑一线信控治理，需：

1. **聚焦四类核心问题**：饱和度、失衡、绿灯空放、溢出；
2. **从「流量与信号配时是否匹配」维度**输出诊断与治理建议；
3. **补齐流量与通行能力数据**，支撑饱和度与配时失配的量化判断；
4. **支持阿里云 ECS 公网部署**，避免 `127.0.0.1` 回环地址导致外网不可达。

---

## 2. 目标用户与场景

| 角色 | 场景 |
|------|------|
| 信控工程师 | 针对某路口晚高峰，快速判断是过饱和、失衡、空放还是溢出，并获得配时调整方向 |
| 运维人员 | 将前后端一键部署到 ECS，通过公网 IP 访问工作台 |
| 智能体/Skill | 将「流量-配时失配」结论固化为可复用 Skill 标签 |

---

## 3. 功能需求

### 3.1 四类问题关注范围

| 类别 | 代码 | 判定依据（现有/补充） | 典型治理方向 |
|------|------|----------------------|--------------|
| 饱和度 | `saturation` | `traffic_flow.saturation_rate`、车道/转向饱和度 | 增加关键方向有效绿灯 |
| 失衡 | `imbalance` | `evaluation.imbalance_index`、转向饱和度极差、`flow_green_verdict` | 重分配绿信比，向高流量转向倾斜 |
| 绿灯空放 | `empty_green` | `evaluation.empty_green_rate`、`green_utilization` | 压缩低利用相位绿灯，转给拥堵方向 |
| 溢出 | `spillback` | `queue_storage_ratio`、`spillback_risk_max`、排队长度 | 控制上游放行，避免排队外溢 |

**要求**：

- 规则引擎支持按上述四类过滤命中规则；
- 诊断流水线输出结构化 `flow_timing_governance` 块，含每类问题的 **是否命中、严重度、证据摘要、治理建议**；
- 与现有 `problem_evidence`、`timing_profile` 数据复用，不重复查库。

### 3.2 流量-配时匹配维度

从以下维度评价「流量与信号配时是否匹配」：

| 维度 | 数据来源 | 输出 |
|------|----------|------|
| 转向流量 vs 有效绿灯占比 | `timing_profile.flow_green_fit`（Spearman τ） | `strong` / `weak` / `mismatch` |
| 车道流量 vs 车道通行能力 | `dws_lane_saturation_5min_mm` + `dws_lane_capacity_5min_mm` | 车道饱和度、能力缺口 |
| 路口整体饱和度 vs 绿信比 | `traffic_flow` + `signal_plan` | 配时是否跟得上需求 |

**治理建议生成原则**：

1. 先判断流量-配时匹配度（匹配 / 偏弱 / 失配）；
2. 再按四类问题给出**可执行**建议（增绿、减绿、重分配、限流/协调）；
3. 若用户有约束（如「垂直方向不能溢出」），约束优先于默认建议；
4. LLM 叙事须引用量化证据，不得脱离数据。

### 3.3 数据补充需求

| 数据项 | 表/字段 | 用途 |
|--------|---------|------|
| 车道流量 | `dws_lane_saturation_5min_mm.lane_flow` | 分车道需求侧 |
| 车道通行能力 | `dws_lane_capacity_5min_mm.lane_capacity` | 饱和度分母、能力瓶颈识别 |
| 车道饱和度 | `lane_flow / lane_capacity` 或表字段 | 精细化过饱和判定 |
| 溢流代理 | `problem_evidence.metrics.spillback_risk_max` | 溢出类问题 |

**降级策略**：无车道能力表时，仅输出路口/转向级指标，并在 `data_gaps` 中标注 `lane_capacity_missing`。

### 3.4 阿里云 ECS 公网部署

| 编号 | 需求 | 验收标准 |
|------|------|----------|
| D1 | 后端监听 `0.0.0.0`，端口可配置 | `curl http://<ECS_IP>:8011/health` 返回 200 |
| D2 | 前端生产构建经 Nginx 反代 `/api`、`/health` | 浏览器访问 `http://<ECS_IP>/` 可完整对话 |
| D3 | 消除硬编码 `127.0.0.1` 生产依赖 | 生产环境变量/compose 不含 loopback |
| D4 | 提供原生一键部署脚本 | `bash scripts/prod-start.sh` 可构建并启动 |
| D5 | 开发环境保持本地回环可选 | `BIND_HOST=127.0.0.1` 时行为与现有一致 |
| D6 | CORS 可配置 | `CORS_ORIGINS` 支持 `*` 或逗号分隔域名 |

**推荐拓扑**：

```
公网 → ECS:80 (Nginx)
         ├─ /        → 前端静态资源
         ├─ /api/*   → backend:8011
         └─ /health  → backend:8011
```

---

## 4. 非功能需求

- 新增服务须有无 DB 单测（mock 数据）；
- 不改变现有 API 契约，仅扩展 `meta.flow_timing_governance`；
- 部署文档含 `.env` 模板与安全组端口说明（80、8011 可选暴露）。

---

## 5. 验收标准

1. 对 mock 过饱和路口，四类诊断至少命中「饱和度」，且 `flow_timing_governance.match_verdict` 有值；
2. `granularity.by_lane` 含 `lane_capacity`（有表时）；
3. ECS 上 `bash scripts/prod-start.sh` 启动后，公网 IP 可访问前端并完成一次 SSE 诊断；
4. 本地 `bash scripts/dev-v2.sh` 仍可用 `127.0.0.1` 开发。

---

## 6. 范围外（本期不做）

- K8s / SLB 多实例编排；
- HTTPS 证书自动申请（可后续加 certbot）；
- 配时优化器自动写回信号机。
