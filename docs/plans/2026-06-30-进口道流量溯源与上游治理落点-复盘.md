## 3. 与计划的偏差与决策

1. **Python 解释器**：环境 `python` 不在 PATH，统一用 `backend/.venv/bin/python` 运行 pytest。
2. **B6 MOCK 拓扑**：可治理上游落点键定为 `dir8==4`（南向），保证北/东/西进口目标在排除来流方向后仍能在 hop2 命中落点；南进口目标不产生落点（可接受）。
3. **B7 进口道识别回退**：MOCK 的 `granularity.by_turn` 行不含 `dir8_code`，新增 `_row_dir8` 按 label 首方位（"东北"先于"东"长匹配）回退，真实数据仍优先用 `dir8_code`。该回退使溯源在 MOCK 主流程也会触发，因此确认文案改为「已定位 N 个上游治理落点，是否生成跨路口协调建议」——对齐计划意图，并据此更新了 `test_api`、`test_skill_fast_path` 两处既有确认文案断言。
4. **F3/F4 装配位置**：播放器状态与侧卡/控制条直接由 `MapStage` 持有并在左下角挂载，控制条事件直接驱动 `MapStage` 既有播放函数，`focus-node` 点选跳到首个揭示该节点的帧。storyboard 自带 `trees.nodes`（含 decision/hop/role/profiles），侧卡无需额外层级数据，避免跨组件 SSE 透传。
5. **容错**：`UpstreamGovernanceTraceService.build` 与 orchestrator `_run_upstream_trace` 均 try/except，溯源失败仅告警、返回空结构，不阻断主诊断。
6. **运镜收敛（map-zoom-debug）**：溯源主视角 zoom 由 13 调整为 **17**；入场一次从渠化拉远后镜头固定，蔓延帧只画线不运镜；换进口树禁止回路口中心；`upstreamEpoch` + `clearMap` 修复回主页残留覆盖物；TTS 仅陈述上游路口事实；理解过程「原因诊断」追加溯源链文案。

## 4. 待手动验证（本环境无法自动执行）

`cd frontend-v2 && npm run dev` 后，触发「奥体西路与经十路交叉口 北进口 早高峰」诊断，核对：

- [ ] 渠化结束后地图平滑过渡到 **zoom ≈ 17**（右下角调试浮层可对照）；
- [ ] 蔓延播放期间镜头**保持不动**，仅渐变连线向外延伸；
- [ ] 换进口（北→东）时**不闪回路口中心**；
- [ ] 回到主页后地图无残留溯源线/点/高亮；
- [ ] 理解过程「原因诊断」出现流量溯源链（`北进口：上游A → 上游B`）；
- [ ] TTS 只播报上游路口名与饱和度，无「治理落点」「共溯 N 个路口」等结论句；
- [ ] 四向饱和环按进口道着色，全饱和节点继续上溯、有空间节点标★落点；
- [ ] 控制条：暂停/单步/进度刮擦拖回可重建、展开二跳折叠、倍速；
- [ ] 多方向 Tab 切换、点选节点不再强制地图跟飞（镜头已固定）；
- [ ] 收束后确认文案为「已定位 N 个上游治理落点…」。

## 5. 后续可优化

- storyboard 体量：二跳 × 多树时帧数/覆盖物增长，关注一次性 payload 体积与高德覆盖物渲染性能。
- 边 `path`：当前真实路径取自 link 几何，缺失退节点直连；可补充沿干线的真实折线以增强叙事。
- 四向饱和环 `DirSaturationRing` 组件目前仅在侧卡/独立测试中使用，地图上的 inspect 环以 badge 近似呈现，后续可在节点 InfoWindow 内嵌该组件。
