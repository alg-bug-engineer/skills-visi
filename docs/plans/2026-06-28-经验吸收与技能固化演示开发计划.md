# 经验吸收可视化 · 技能固化演示开发计划

> **版本**：2026-06-28 v2（L3 交错 + 左抽屉）  
> **状态**：已实施（2026-06-28 v2）  
> **目标**：面向领导汇报演示——在用户确认固化后，以**系统自言自语式追踪流**呈现智能体对一线民警经验的吸收、检索、比对与转化过程，并与 Skill 落盘**同框交错**（L3）；总时长 **≤40 秒**；体现「边想边写」与「众智成城」。  
> **关联文档**：[技能沉淀与匹配逻辑开发计划.md](./技能沉淀与匹配逻辑开发计划.md)、[PROJECT_OVERVIEW.md](../PROJECT_OVERVIEW.md)、[backend/docs/API.md](../../backend/docs/API.md)、[backend/docs/PROJECT_LOGIC.md](../../backend/docs/PROJECT_LOGIC.md)

## 1. 背景与演示定位

### 1.1 演示给谁看

| 受众 | 要感知到的价值 |
|------|----------------|
| **领导** | 有了智能体，最基层一线民警的口述经验能被系统吸收、入库、复用；「众智成城」，而非困在专家/民警脑子里的经验孤岛 |
| **一线民警（旁观）** | 自己的话被系统「听懂并存档」，下次不必重复 |
| **技术人员（旁观）** | 吸收过程每个数字可追溯到 SSE payload，非表演 |

### 1.2 与现有能力的关系

系统已具备完整诊断链路与 Skill 落盘能力。缺口在于：

- 用户点「确认固化」后，**直接进入** `skill_build` 代码打字，缺少「经验被消化」的可视阶段；
- 领导无法一眼看懂：**这次沉淀了什么、有什么好处、与未沉淀时的诊断逻辑有何差异**。

本计划**不替换**现有落盘逻辑，在 `upsert_from_session_visual` 内以 **L3 交错可视化器**统一编排吸收与写入。

### 1.4 v2 演示节奏（领导彩排）

| 阶段 | 时长预算 | 视觉 |
|------|----------|------|
| 头脑追踪 recap → decompose → retrieve → compare → value | ~27s | 右栏 `ExperienceAbsorptionPanel` |
| blueprint 引导 + `write_phase_start` | ~2s | 右栏 |
| L3 交错写文件（每文件：右栏联动行 + 左抽屉打字） | ~11s | 左 `SkillBuildDrawer` + 右栏同步 |
| packaging + done | ~1s | 抽屉完成态 → 自动收起 |
| **合计** | **≤40s** | `demo_pacing.py` 控速 |

### 1.5 v2 布局原则

- **左抽屉**：叠在 `stage-body` 地图区域左侧，终端风格，**非全屏**；
- **右栏**：吸收面板始终可见，**不被抽屉遮挡**——同框体现「边想边写」；
- **取消** v1 全屏 `SkillBuildOverlay`。

### 1.3 项目约束（必须遵守）

| # | 约束 | 说明 |
|---|------|------|
| C1 | **忽略 `frontend/`** | v1 为历史独立仓库，禁止读取、修改、部署 |
| C2 | **治理建议及之前步骤尽量不动** | NLU → 路口匹配 → 认知 → 查数 → 问题验证 → 规则 → 建议二次确认，编排与 SSE 保持不变 |
| C3 | **`skill_build` 落盘过程保留** | `SkillBuildVisualizer` 写文件、diff、下载逻辑为第二幕「物证」，不可删减 |
| C4 | **真实数据原则** | 吸收过程全部来自 session、技能库检索、`diff_with_session`、标签构建等**真实计算**；禁止前端或 LLM 即兴编造「好处文案」 |
| C5 | **系统自言自语，禁止旁白讲解** | 文案为主语省略的系统日志体；禁止「接下来为您介绍」「众智成城意味着」等教学口吻 |
| C6 | **触发时机** | 用户回复「是」确认固化（`skill_action=awaiting_create` 之后）才进入吸收阶段 |
| C7 | **回顾 + 增量** | 吸收阶段不重复诊断已展示的证据/约束长文，仅极简回顾 + 检索/比对/价值/转化等**增量**信息 |

## 2. 总体架构

### 2.1 流水线（L3 交错）

```
… → 生成治理建议 → 用户确认固化「是」
                        ↓
              ┌──────────────────────────────────────┐
              │ InterleavedSkillPersistVisualizer     │
              │ ① skill_absorption 头脑阶段（paced）   │
              │ ② write_phase_start + drawer_open     │
              │ ③ 每文件：吸收 L3 行 ∥ skill_build 打字 │
              │ ④ drawer_close + absorption_done      │
              │    + skill_build_done                 │
              └──────────────────────────────────────┘
```

**后端入口**：`SkillService.upsert_from_session_visual()` → `InterleavedSkillPersistVisualizer.emit()`（替代 v1 串行 `ExperienceAbsorptionVisualizer` + `SkillBuildVisualizer`）。

**编排层**：`Orchestrator._persist_skill_from_session()` 不变。

### 2.2 同框分工（v2）

| 区域 | SSE | 气质 |
|------|-----|------|
| 右栏吸收 | `skill_absorption` | 冷色日志 · 检索/比对/价值 + L3 文件联动行 |
| 左抽屉 | `skill_build`（`drawer_open`…`drawer_close`） | 终端 IDE · 文件树 + 打字落盘 |

## 3. UI 方案（v2 · 右栏 + 左抽屉同框）

### 3.1 布局结构

```
┌─ stage-column ─────────────────────┐ ┌─ process-column ────┐
│ ┌ SkillBuildDrawer（左，终端）─┐   │ │ 上：理解过程(折叠)   │
│ │ 文件树 │ 打字区              │   │ │ 下：经验吸收 trace   │
│ └──────────────────────────────┘   │ └─────────────────────┘
│        地图（中间仍可见）            │
└────────────────────────────────────┘
```

| 行为 | 说明 |
|------|------|
| 吸收开始 | 右栏 `panelLayout=stacked`，理解过程折叠为摘要 |
| `drawer_open` | 左抽屉自地图左侧滑入，宽度约 52%，**不覆盖**右栏 |
| L3 写文件 | 每写一文件：右栏输出 `render_file_linkage_line`，左抽屉同步 `file_delta` |
| `skill_build_done` | 抽屉显示完成 + 下载链接，约 2.2s 后自动收起 |
| 会话重置 | `initSession` 同步 `resetSkillBuild` / `resetExperienceAbsorption` |

### 3.2 与地图的关系

- 抽屉仅占 stage-column 左侧，地图右侧区域保持可见；
- 右栏 `process-column` 独立列，与抽屉无 z-index 冲突。

### 3.3 废弃项（v1）

- ~~全屏 `SkillBuildOverlay`~~ → 由 `SkillBuildDrawer` 替代。

### 3.3 视觉规范

| 元素 | 理解过程（现有） | 经验吸收（新增） |
|------|------------------|------------------|
| Eyebrow | `ANALYSIS` | `ABSORPTION` |
| 字体 | 正文时间线 | 等宽/半等宽日志体 |
| 主色 | 现有青蓝 | 低饱和冷灰 + 状态色（running=琥珀，done=绿） |
| 内容形态 | 步骤标签 + 推理句 | Trace 行 + Evidence Chips |
| 打字速度 | ~22ms/字 | ~12–15ms/字（更快，日志感） |

### 3.4 自言自语文案规范（后端模板生成）

**允许：**

```
> 收到固化指令。
> 挂载会话摘要：「会展路与奥体中路，晚高峰南北向拥堵…」
> 解构 tags.match：inter_id=011wwe29jbf00001，period=晚高峰，directions=[南北向]
> 扫描技能库：总记录 12 条，match 过滤后 0 条。
> 槽位 skill_011wwe29jbf00001_congestion_晚高峰 空闲。
> 判定：CREATE。经验库计数 12 → 13。
> 吸收完成。开始写入技能包目录。
```

**禁止：**

```
> 接下来为您介绍本次沉淀的经验。
> 众智成城意味着基层智慧将被汇聚……
> 您将获得以下三大好处……
```

所有 `>` 行由后端 `ExperienceAbsorptionRenderer` 用**模板 + 真实字段**拼接；前端**原样流式显示**，不得追加文案。

在现有 `build_skill_tags()` 基础上扩展，写入 `skill.meta.json` 的 `tags` 字段（向后兼容 `backfill_tags`）。

### 4.1 三层结构

```json
{
  "match": {
    "inter_id": "011wwe29jbf00001",
    "intersection": "会展路与奥体中路路口",
    "time_period": "晚高峰",
    "problem_type": "congestion",
    "directions": ["南北向"],
    "match_keywords": ["拥堵", "晚高峰", "南北向"]
  },
  "content": {
    "rule_ids": ["R-SAT-001"],
    "issue_codes": ["saturation", "imbalance"],
    "constraint_intent": "max_green_cap",
    "has_user_constraints": true,
    "suggestion_formula_hash": "a1b2c3…",
    "data_window_profile": "dwd_7d_weekday_evening"
  },
  "meta": {
    "experience_source": "field_officer",
    "source_utterance_summary": "会展路晚高峰南北向堵，垂直方向不能溢出",
    "contributor_role": "一线民警",
    "absorbed_at": "2026-06-13T10:32:07+08:00",
    "version": 1,
    "library_count_before": 12,
    "library_count_after": 13
  }
}
```

### 4.2 字段来源（保证真实）

| 字段 | 来源 |
|------|------|
| `match.*` | `session.nlu` + `session.inter_id` + `_extract_keywords` |
| `content.rule_ids` | `session.diagnosis.matched_rules` |
| `content.issue_codes` | `session.data_payload.flow_timing_governance.issue_codes`（无则省略） |
| `content.constraint_intent` | `quantitative_constraints.intent` |
| `content.suggestion_formula_hash` | `sha256(suggestion_formula)[:12]` |
| `content.data_window_profile` | `data_query_spec.data_window` 摘要 |
| `meta.source_utterance_summary` | `session.raw_user_context` 截断 80 字 |
| `meta.library_count_*` | `len(skill_service.list_skills())` 吸收前后 |

### 4.3 检索与比对（吸收阶段调用）

| 操作 | 实现 | 输出给 UI |
|------|------|-----------|
| 检索 | 复用 `match_skill()` + 全库 `list_skills()` 计数 | `retrieve.total`, `retrieve.matched`, `retrieve.candidates[]` |
| 比对 | `diff_with_session(existing, session)` + 扩展 content 层字段 diff | `compare.action`, `compare.changes[]` |
| 判定 | `existing is None` → CREATE；`has_material_diff` → UPDATE；`_snapshot_equal` → UNCHANGED | `compare.action` |

**不修改**快路径 `find_match_result` 的匹配行为；吸收阶段只读调用。

### 5.3 SSE 事件契约

顶层 `event` 固定为 `skill_absorption`。`type` 与 `stage` 如下：

| type | stage | 时机 | payload 关键字段 |
|------|-------|------|------------------|
| `skill_absorption_start` | — | 开始 | `session_id`, `skill_id`, `intersection` |
| `stage_start` | `recap` | 回顾 | `display_text`, `utterance_summary` |
| `thought_delta` | `recap` | 流式日志 | `delta` |
| `stage_done` | `recap` | | `duration_ms` |
| `stage_start` | `decompose` | 标签解构 | `tags`（完整三层） |
| `evidence` | `decompose` | 芯片数据 | `chips: [{key, label, value}]` |
| `stage_done` | `decompose` | | |
| `stage_start` | `retrieve` | 库检索 | `total_skills`, `query` |
| `evidence` | `retrieve` | | `candidates: [{skill_id, created_at, tags.match}]` |
| `stage_done` | `retrieve` | | `matched: bool`, `hit_count` |
| `stage_start` | `compare` | 比对 | `action`: CREATE\|UPDATE\|UNCHANGED |
| `evidence` | `compare` | | `changes: string[]`, `before`, `after` |
| `stage_done` | `compare` | | |
| `stage_start` | `value` | 价值事实表 | `what`, `why_rows`, `delta_rows` |
| `stage_done` | `value` | | `library_count_before`, `library_count_after` |
| `stage_start` | `blueprint` | 转化蓝图 | `files: string[]`, `package_dir` |
| `stage_done` | `blueprint` | | |
| `skill_absorption_done` | — | 结束 | `action`, `skill_id`, `progress: 100` |

**节奏**：每阶段 `stage_start` → 若干 `thought_delta`（由 renderer 预生成行列表）→ `evidence`（若有）→ `stage_done`。`duration_ms` 为真实耗时。禁止纯 `asyncio.sleep` 凑时长；仅保留 ≤50ms 的 UI 可读性微延迟。

### 5.4 价值阶段 `value` 数据结构（领导一眼看懂）

全部由后端计算，示例：

```json
{
  "what": {
    "title": "会展路与奥体中路路口 · 晚高峰 · 南北向拥堵",
    "bullets": [
      "场景键：inter_id + 晚高峰 + 南北向",
      "判别规则：R-SAT-001",
      "治理边界：max_green_cap（来自民警约束）",
      "查数口径：dwd_7d_weekday_evening"
    ]
  },
  "why_rows": [
    {"key": "retrievable", "before": "false", "after": "true"},
    {"key": "fast_path", "before": "不可用", "after": "skill_011wwe29jbf00001_congestion_晚高峰"},
    {"key": "constraint_snapshot", "before": "仅本次会话", "after": "写入 tags.content"},
    {"key": "library_entries", "before": "12", "after": "13"}
  ],
  "delta_rows": []
}
```

`why_rows` 为**事实对照**，不用形容词；领导自行得出「众智成城」结论。

### 5.5 CREATE / UPDATE / UNCHANGED 分支

| action | 吸收叙事差异 | `skill_build` |
|--------|--------------|---------------|
| **CREATE** | 检索 0 命中 → 槽位空闲 → 库计数 +1 | 正常 `skill_build_start` |
| **UPDATE** | 命中已有 skill → 展示 `diff.changes` → version+1 | `is_update=true`, `diff_changes` |
| **UNCHANGED** | 压缩为 3–5 行日志 → 判定 UNCHANGED | 现有 `unchanged` 快速 `skill_build_done`，可跳过打字 |

### 6.4 `useExperienceAbsorption` 状态模型

```typescript
interface AbsorptionTraceLine {
  seq: number
  stage: AbsorptionStage
  kind: 'monologue' | 'stage_marker' | 'evidence'
  text?: string           // 仅来自 SSE thought_delta / display_text
  chips?: EvidenceChip[]  // 仅来自 SSE evidence
  status: 'running' | 'done'
  durationMs?: number
}

interface ExperienceAbsorptionState {
  active: boolean
  currentStage: AbsorptionStage | 'idle'
  lines: AbsorptionTraceLine[]
  valueSnapshot: ValuePayload | null
  action: 'create' | 'update' | 'unchanged' | null
  progress: number
}
```

**禁止**在 reducer 内根据 `action` 拼接教学文案。

### 6.6 与 SkillBuildDrawer 的时序（v2）

1. `skill_absorption_*` 在右栏下部展示；
2. `write_phase_start` → `drawer_open` → `SkillBuildDrawer` 自地图左侧滑入（非全屏）；
3. 每写一文件：右栏 `render_file_linkage_line` + 左抽屉 `file_delta` 同步；
4. `drawer_close` → `skill_build_done` → 抽屉约 2.2s 后自动收起 → `initSession` 重置两层面板。

### 6.7 ProcessTimeline 第 8 步

`skill_absorption_start` 时将「技能固化」步标为 `running`；`skill_build_done` 标为 `done`。吸收与写盘同属第 8 步，不在理解过程重复 enqueue 长文（避免与吸收 Trace 重复）。

## 8. 领导彩排脚本

设置 `DEMO_MODE=1`，锚定日主秀路口。

### 8.1 第一场 — CREATE（众智成城 0→1）

1. 输入：「会展路与奥体中路，晚高峰南北向拥堵，垂直方向不能溢出」
2. 走完诊断 → 生成建议 → 确认固化
3. **预期吸收日志**：检索 0 命中 → CREATE → 库计数 N→N+1
4. **预期 skill_build**：新建包目录打字
5. **领导话术**（人讲，系统不讲）：「民警刚才那句话，进了城市经验库。」

### 8.2 第二场 — UPDATE（经验迭代）

1. 同路口同场景，补充新约束或触发规则变化
2. 全量诊断后确认固化
3. **预期**：命中已有 skill → diff 行 → UPDATE → version 2
4. **领导话术**：「一线补充的边界，系统更新了城市记忆。」

### 8.3 第三场 — 快路径（吸收结果兑现）

1. 同场景无新约束再次提问
2. **预期**：诊断阶段 `skill_match` 命中（非吸收阶段）
3. **领导话术**：「上次沉淀的经验，这次直接复用。」

## 10. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 吸收与 skill_build 总时长过长 | 吸收阶段真实计算为主，微延迟 ≤50ms/行；UNCHANGED 走短路径 |
| 文案滑向旁白 | 文案只许后端 renderer 输出；CR 检查禁用「您」「我们」 |
| 标签扩展破坏旧 Skill | `backfill_tags` + 匹配逻辑只读 match 层 |
| 右栏过高内容挤占 | 上部折叠条固定 ~40px；下部 `overflow-y: auto` |
| 理解过程与吸收内容重复 | 吸收 recap 仅 1 行摘要；证据不在吸收重复长文 |
