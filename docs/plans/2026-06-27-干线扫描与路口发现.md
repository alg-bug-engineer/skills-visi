> 版本：2026-06-29  
> 状态：已落地（backend + frontend-v2）  
> 单测：118 passed（含干线/意图/选型/几何）

---

## 1. 业务目标

在**单点路口诊断**之前，支持交警以自然语言询问一条道路上的拥堵分布，例如：

- 「奥体西最拥堵的路口是哪个？」
- 「奥体西路早高峰哪些路口比较堵？」
- 「奥体西晚高峰经常拥堵的路口有哪些？」

**设计原则（产品约束）**

| 原则 | 说明 |
|------|------|
| 不自动默认 Top1 | 扫描结果全量路口上图，用户主动选型 |
| 扫描三要素 | 干线 + 时段 + 拥堵；缺时段必须追问 |
| 排名确定性 | 饱和度等由 PG + 公式排名；LLM 只写解说与引导 |
| 选型后衔接 | 用户选定路口 → 注入单点 NLU，**只追问方向**，再走现有诊断流水线 |

## 2. 会话状态机（新增/变更）

```
IDLE / DONE
  ├─ 首轮意图 = corridor_scan → CORRIDOR_SCANNING → … → AWAITING_CORRIDOR_PICK
  └─ 首轮意图 = intersection_diagnosis → PROCESSING（原单点流水线）

CORRIDOR_NLU_INCOMPLETE   # 缺 corridor 或 time_period，追问（不问具体路口名）
AWAITING_CORRIDOR_PICK    # 扫描完成，等用户选路口或说「最堵的/第二个」
  └─ 选型成功 → NLU_INCOMPLETE（仅 directions）或 PROCESSING
```

**ReplyType 新增**：`corridor_scan`（扫描结果 + 地图场景 + Top3 文字引导）

## 3. 首轮意图识别（LLM + 规则兜底）

### 3.1 流程

1. **会话状态优先**：`awaiting_corridor_pick` / `corridor_nlu_incomplete` 不走 LLM
2. **LLM 二分类**（`IntentClassifierService`）：`corridor_scan` | `intersection_diagnosis`
   - `enable_thinking=False`，`max_tokens=80`，`max_retries=0`，低温
3. **规则兜底**（`intent_router.route_intent_by_rules`）：LLM 失败或 `confidence=low` 时启用

### 3.2 规则要点（与提示词对齐）

**干线扫描（corridor_scan）**

- 哪些/哪几个/有哪些 + 路口；最堵/最拥堵；哪里堵；经常拥堵 + 路名
- 道路名 + 拥堵 + 复数/位置疑问（如「路口有哪些」倒装也支持）

**单点诊断（intersection_diagnosis）**

- 已出现「X路与Y路」具体路口名 + 方向/成因/治理
- 默认分支

**排除**

- `extract_intersection_phrases` 命中标准「X路与Y路」且无扫描关键词 → 单点

### 3.3 关键模块

| 模块 | 路径 |
|------|------|
| LLM 分类 | `backend/intersection_agent/services/intent_classifier_service.py` |
| 规则兜底 | `backend/intersection_agent/services/intent_router.py` |
| 编排入口 | `orchestrator.handle_message` → `_run_corridor_pipeline` |

## 4. 干线扫描流水线

```
用户输入
  → CorridorScanNluService（LLM 提取 corridor + time_period）
  → LineResolver（dim_line_info 区段 / road_intersections 降级）
  → CorridorScanService（批量 dws_inter_evaluation_5min_mm 排名）
  → CorridorNarrativeService（Top3 解说 + 选型引导）
  → build_corridor_scan_scene（地图 payload）
  → 响应 meta：corridor_scan + corridor_scan_scene
```

### 4.1 干线解析（LineResolver）

- 口语「奥体西」→「奥体西路」
- 优先 `line_name LIKE '奥体西路%'` 且 `inter_count` 最大的 `dim_line_info` 区段
- 无 line 区段时降级：`dim_inter_info.inter_name LIKE '%路名%'`（`scan_mode=road_intersections`）

### 4.2 排名公式

`corridor_metrics.congestion_score`：以 `saturation_max` 为主，辅以 `unbalance_index` / `delay_index`

### 4.3 路网几何（可视化）

**问题（已修复）**：`dim_line_link_rltn.seq_no` 与地理顺序不一致，逐 link 绘制会产生多条平行直线。

**做法**：

1. 按 `dim_line_inter_rltn` **路口顺序**，用 link 的 `f_inter_id→t_inter_id` 串联几何；缺段 BFS 补全
2. 合并为**一条** `polyline`；路口坐标 snap 到中心线
3. 前端只绘制合并 polyline，不逐 link 叠线

| 模块 | 路径 |
|------|------|
| 几何链化 | `backend/intersection_agent/utils/corridor_geometry.py` |
| 扫描服务 | `corridor_scan_service.py` |
| 场景构建 | `corridor_scan_scene.py` |

## 5. 路口选型（Corridor Pick）

用户处于 `awaiting_corridor_pick` 时可：

- 「最拥堵的 / 第二个 / 第三个」
- 口语路口名：「奥体西与经十路」→ 匹配「奥体西路与经十路路口」（路名规范化 + A与B 双路匹配）
- **前端**：左侧列表 / 地图 pin 点击 → 自动发送完整 `inter_name`

选型成功后：

1. 写入 `session.nlu`（路口 + 扫描时段）
2. 若缺 `directions` → 追问方向
3. 否则进入 `_diagnose` 单点流水线

| 模块 | 路径 |
|------|------|
| 选型解析 | `corridor_pick_resolver.py` |
| 路名工具 | `corridor_geometry.normalize_road_token` / `extract_junction_roads` |

## 9. 语音播报（同分支 feature/tts 摘要）

- TTS 由阿里云 ISI 迁移为 **Qwen-TTS Realtime**（WebSocket PCM）
- 关键点引导式播报；前端 `pcmStreamPlayer` + `useVoiceNarration`
- 配置见 `backend/.env.example`：`QWEN_TTS_*`、`TTS_ENABLED`

## 10. 已知限制

1. `road_intersections` 降级模式无路网中心线时，仅路口点 + 无 link 链化
2. 多条 `dim_line_info` 区段同名路时，当前取 `inter_count` 最大的一段
3. 意图 LLM 仅在首轮调用；后续轮次靠会话状态机
