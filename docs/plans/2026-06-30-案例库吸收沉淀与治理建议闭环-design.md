# 案例库吸收沉淀与治理建议闭环 — 设计

- 日期：2026-06-30
- 分支：`feat/case-library-and-absorption-loop`
- 目标：把「专家行业案例 + 历史路口案例」系统化为可浏览的案例库；让用户提供的认知/诊断经验在验证阶段以右下角浮层卡片显式反馈吸收结果（含去重）；治理建议严格参考案例，并标注可溯源、可跳转的案例/经验依据，形成「案例 → 经验 → 诊断 → 建议 → 回溯案例」的完整闭环。

## 背景与现状（复用，不重建）

- `IntersectionProfileStore`：每路口一个 JSON，三桶 `cognition / diagnosis / solution_ref`，已含归一化去重与状态升级（`verified > data_doubt > manual`）。
- `CaseLibraryService`：已解析 `docs/expert_knowledge.md`（1314 案例 / 19 场景）→ 场景/典型问题/治理方案，已用于喂给大模型生成建议（`meta.case_experience`）。
- `/experience/library`：聚合三桶 → 现有「经验库」Tab 的 认知/诊断/方案 三个子 Tab。
- SSE 事件 `experience_cognition / experience_diagnosis / experience_solution`（orchestrator `_record_*`）：已带 `text/status/tags`，前端据此累积 `experienceSediment`。
- `UnderstandingProcessPanel.vue`：承载「理解过程 / 经验库」Tab。
- `WorkbenchLayout.vue`：工作台布局，含 `ExperienceAbsorptionPanel`、`map-toast`。

## 需求 1：新增「案例库」Tab（行业案例 / 路口案例）

### 放置
`UnderstandingProcessPanel.vue` 新增第三个顶层 Tab `cases`（与 理解过程/经验库 并列）。案例库内含两个子 Tab：**行业案例**、**路口案例**。新组件 `CaseLibraryPanel.vue`。

### 行业案例（结构化 expert_knowledge.md）
- 后端：`CaseLibraryService` 增 `list_all()`，完整解析每场景的：场景名/ID/描述、典型问题（出现频次、典型表现）、治理方案（频次、关键措施、适用条件、注意事项、代表案例 `#N` 标题+摘要）。当前 `_parse` 丢弃了频次与代表案例，需增强解析。
- 新路由 `GET /api/v1/cases/industry` → 返回结构化场景数组（可选 `?scenario_id=`、`?q=` 过滤）。新 Pydantic 模型。
- 稳定 ID：场景级 `industry:{scenario_id}`；代表案例级 `industry:case:{n}`。
- 前端：左侧场景列表（含案例数徽标）→ 选中展开典型问题 → 治理方案卡（措施/适用/注意 + 代表案例可读摘要）。顶部按场景名/问题关键词搜索。

### 路口案例（历史路口诊断→治理方案，复用现有档案聚合）
- 后端：新路由 `GET /api/v1/cases/intersections`，由 `IntersectionProfileStore.load_all()` 组装，**仅保留 `solution_ref` 非空**（已形成治理方案）的路口为一个案例。
- 每案例结构（仿专家案例可读性）：`inter_id / 路口名 / 时段`、`场景认知`（cognition 文本+状态）、`诊断问题与成因`（diagnosis cause/dimension/confidence）、`治理方案与量化成效`（solution_ref → skill 富化的 measure/qualitative/quantified/download_url）、`沉淀时间`。
- 前端：路口案例子 Tab 渲染为结构化卡（背景 → 诊断 → 方案 → 成效），与行业案例排版统一。

## 需求 2：经验验证浮层 Toast（右下角向上弹出，含去重反馈）

### 组件
新建浮层组件 `ExperienceAbsorptionToast.vue`，挂在 `WorkbenchLayout.vue`（`position: fixed; right/bottom`，slide-up 进场，自动消隐 + 可手动关闭，支持队列堆叠）。配套 `useAbsorptionToasts.ts` 管理队列。

### 触发与文案（复用现有 SSE 事件，增补 `action` 去重结果）
后端三个 `_record_*` 的 store 方法（`add_cognition/add_diagnosis/add_solution_ref`）改为返回吸收结果枚举 `inserted | exists | updated`，并把 `action` 加进对应 SSE 事件 `data`。

- **认知经验**（用户提供，如「早高峰有空放」）：在获取运行数据并验证后触发。
  - 头部：有数据支撑 → 「已验证用户提供的路口认知经验」；无验证数据 → 「用户提供的路口认知经验未找到数据支撑，已记录」。
  - 主体：标签化的经验理解（认知画像 / 问题记录 / 已验证·待验证 + 关键词）。
  - 尾部：`已插入 / 已存在 / 已更新` 该路口认知经验库（更新逻辑＝状态升级或补 evidence；exists＝命中且无变化）。
- **诊断经验**（用户口述，可能无数据，如「附近四点放学 / 常有演唱会 / 早上有特勤」）：在问题验证阶段触发。
  - 头部：「已吸收用户提供的诊断经验并入库」。
  - 主体：结构化诊断经验标签。
  - 去重：命中已存在 → 「该诊断经验已存在（不再插入/更新）」。
- **方案经验**：沿用当前技能固化流程（固化完成可给一条提示型 toast，非本需求重点）。

前端：在 SSE 分发处把 `experience_cognition/diagnosis` 事件（带 `action/status/tags`）推入 toast 队列。

## 需求 3：治理建议参考案例 + 可溯源闭环

- 后端：`SuggestionResult` 增 `references: list[SuggestionReference]`，每条 `{ type: 'industry'|'intersection', id, scenario_id?, title, summary }`。由 orchestrator 已算出的 `case_matches`（专家场景）+ `reused_experience`（本路口历史经验）确定性派生，作为该建议块的依赖来源；随 `suggestion` 与 `meta` 下发。
- 前端：治理建议渲染处（`IntersectionNarrativeStack.vue` 建议卡）追加「参考依据」chip 列表：
  - hover：tooltip 显示 概要 + 来源类型 + ID。
  - 点击：跳转「案例库」Tab 并定位到对应条目（行业案例锚到场景/代表案例；路口案例锚到对应路口）。
- 形成闭环：案例库（来源）↔ 治理建议（引用）双向可达。

## 数据流

```
expert_knowledge.md ──CaseLibraryService.list_all()──▶ /cases/industry ──▶ 案例库·行业案例
IntersectionProfileStore.load_all()(solution_ref≠∅) ──▶ /cases/intersections ──▶ 案例库·路口案例
用户原话 ─classify→ add_cognition/diagnosis(返回 action) ─SSE experience_*─▶ 右下角 Toast
case_matches + reused_experience ──▶ SuggestionResult.references ──hover/click──▶ 案例库
```

## 测试要点

- 后端：`CaseLibraryService.list_all()` 解析频次/代表案例；`/cases/industry`、`/cases/intersections` 路由；store 方法返回 `inserted/exists/updated`（含状态升级=updated、命中无变化=exists）；`SuggestionResult.references` 派生。
- 前端：案例库三态（行业/路口/空）渲染；toast 队列对 cognition(verified/data_doubt × inserted/exists/updated) 与 diagnosis(inserted/exists) 文案分支；建议 reference chip hover/跳转。
- 回归：现有 经验库 Tab、`/experience/library`、建议生成、`meta.case_experience` 不破坏。

## YAGNI / 不做

- 不引入向量检索；行业案例匹配维持关键词。
- 路口案例不新建独立存储，复用 profile 档案聚合。
- 方案经验 toast 不做复杂更新逻辑。
