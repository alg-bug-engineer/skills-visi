## 2. 核心能力说明

### 2.1 主诊断 `primary_diagnosis`

`FlowTimingGovernanceService._build_primary_diagnosis()` 输出四类之一：

| type | 含义 |
|------|------|
| `timing_optimizable` | spread 大 + 高饱和 → 绿灯分配不均，配时可改善 |
| `capacity_bottleneck` | spread 小 + 普遍高饱和 → 能力瓶颈，出路在非配时手段 |
| `structure_limited` | 叠加标注：关键转向触最小绿 |
| `basically_matched` | 供需与配时基本匹配 |

`summary` / `governance_narrative` 以 `headline` 起句；四维 `problems[]` 退为补充扫描。

### 2.2 结构化动作方案 `action_plan`

`GovernanceActionPlanService.build_action_plan()` 按证据推导：

- `action_type`：`reallocate_green` / `increase_green` / `capacity_non_timing` / `spillback_control` / `maintain` / `guidance_only`
- `transfer_seconds`：基于转向饱和度、绿灯利用率、流量/绿信比份额，**非** `sat×12` 公式
- `donor_turn` / `recipient_turn`：供绿方与受绿方
- `narrative_template`：LLM 只可润色、不可改数

优先级：溢流严重 → 能力瓶颈 → 转向级挪绿 → 定向加绿 → 四维文案兜底。

### 2.3 专家规则 YAML

`governance_guidance.py` 加载 `governance_rules.yaml`，按 `primary_type`、四维命中、spread、利用率等条件匹配分维度治理文案，供 `SuggestionService` prompt 注入。

### 2.4 用户约束合并

`user_constraint_merge.merge_user_constraints()`：按子句去重合并用户治理约束，用于 Skill 快路径与 orchestrator 多轮补充。

### 2.5 术语去数据仓化

`problem_evidence_service` 等方法字段映射为业务语言；用户可见 narrative 不再暴露 DWS/DWD 表名。

### 2.6 前端呈现

- `FlowTimingGovernanceCard`：头牌 `headline` + `lever` + severity 配色；`action_plan` 摘要
- `channelizationCopy`：渠化图右上角与主诊断类型联动
- `waitForGovernanceSuggestionPresented`：治理建议呈现栅栏，对齐语音/叙事节拍
- `App.vue` / `WorkbenchLayout`：治理段 TTS 以 `primary_diagnosis.headline` 起句
