## 0. 数据事实（实测，作为设计基线）

对 `docs/all_slices_dws_correlate_m_multihop_merged.csv`（978,822 行）实测：

| 事实 | 数值 | 设计含义 |
|------|------|----------|
| 唯一基准路口 | 273 | 覆盖面足够 |
| period 分布 | 早 32.8w / 晚 33.8w / 平 31.3w | 与 NLU 三时段一一对应 |
| trace 分布 | UPSTREAM 48.8w / DOWNSTREAM 49.1w | 双向齐备，本期只用 UPSTREAM |
| 演示路口 `011wwe28ctu00001` | 基准 UPSTREAM 2050 行 | **有数据**，可做主秀 |
| **组内 `flow_share_ratio` 之和** | **中位 1190%，最大 3001%** | ⛔ 不是「占比」，是**路径途经率** |
| share 恰好=100% 行 | 19,224（2%） | 直接上游/必经节点 |

**`flow_share_ratio` 正确语义**：本路口某转向的车流中，有 X% **途经过**该上游转向（多跳叠加，故可 >100%）。
→ 同一上游方位（`cor_f_dir8_no`+`cor_turn_dir_no`）上，share 随距离单调衰减，天然形成「跳数链」。

实测演示路口「北进口直行」UPSTREAM 同方位链：`90.5% → 76.7% → 47.7% → 42.4%`（即一跳→二跳→三跳→四跳）。

## 1. 指标重定义（任务①）

废弃旧设计中的 `top1_share≥35%`、`top3累计≥70%`（基于「占比可归一到 100%」的错误前提）。
新口径全部基于**路径途经率的相对结构**，不做跨来源求和。

### 1.1 术语与计算

| 名称 | 定义 | 计算 |
|------|------|------|
| `path_coverage` | 单条来源的途经率 | = `flow_share_ratio`（0–300+） |
| **一跳节点** | 每个上游方位的最近节点 | 同 `(cor_f_dir8_no, cor_turn_dir_no)` 分组内 `path_coverage` 最大者（见 §1.2 一跳锁定） |
| `feed_direction` | 来车方位 | 由 `cor_f_dir8_no` 映射为「东/南/西/北 进口」+ `cor_turn_dir_no` 转向 |
| `dir_coverage[dir]` | 某来车方位的主导途经率 | 该方位**一跳节点**的 `path_coverage` |
| `dominant_feed` | 主导来车方位 | `dir_coverage` 最大的方位 + 其一跳路口 id/name |
| `dominance_gap` | 主导显著度 | top1 `dir_coverage` − top2 `dir_coverage`（绝对差） |

### 1.2 一跳锁定（任务③）

表无 hop 字段，按以下优先级锁定一跳，**本期 MVP 采用 B**：

- **方案 A（权威，二期）**：join `road6.dwd_tfc_rltn_wide_inter_ft_link` 相邻关系，仅保留直接相邻 `cor_inter_id`。
- **方案 B（数据自洽，MVP）**：对每个 `(my f_dir8, my turn)` 的 UPSTREAM 集合，按 `(cor_f_dir8_no, cor_turn_dir_no)` 方位分组，**每组取 `path_coverage` 最大的一行**作为该方位一跳节点；其余视为多跳，丢弃。

> 方案 B 不依赖额外表，且实测衰减链证明「同方位最大 share=最近上游」成立。P0 验收需人工核对一跳节点确在地理上游。

### 1.3 来源结构判定（驱动治理）

仅在「问题转向」上计算（问题转向见 §2.1）。阈值入 `rules/thresholds_extensions.yaml`，演示数据校验后定稿：

| 判定 `source_pattern` | 条件 | 治理落点 |
|------|------|----------|
| `single_corridor` 单一上游主导 | top1 `dir_coverage` ≥ `coverage_high`(初值 0.70) 且 `dominance_gap` ≥ `gap_significant`(初值 0.25) | 建议在 `dominant_feed` 的一跳路口做信控/协调 |
| `multi_corridor` 多向汇入 | ≥2 个方位 `dir_coverage` ≥ `coverage_high` | 本路口+多上游联控 / 区域协调 |
| `local` 无显著外部来源 | 所有 `dir_coverage` < `coverage_high` | 回到本路口配时/渠化，不外溢建议 |

### 1.4 文案口径（强制）

- ✅ 「本进口约 {cov}% 的车流**途经**上游{路口}{方位}」
- ❌ 「{路口}**贡献**了 {cov}% 流量」（语义错误，禁止）
- 月级口径必须标注：「依据近月同时段出行规律」。

## 2. 接入位置与四链路对齐（任务④）

### 2.1 流水线位置

复用单点 `_diagnose()` 现有顺序，溯源作为**问题诊断环节的增强层**，不新增第 9 步：

```
data_fetch(STEP3) ──→ 新增 _fetch_flow_correlate（轻量、条件触发）
problem_evidence(STEP4) ──→ diagnosis_story 增 phase=flow_trace beat（思考过程）
rule_engine + flow_timing_governance(STEP5) ──→ 主诊断 hints + action_plan 落点
治理建议(STEP6) ──→ action_type=upstream_coordination 文案带路口名
```

**问题转向**（溯源只在其上算，避免全表罗列）：
```
problem_turns = {NLU关注方向} ∩ {by_turn 中 turn_saturation ≥ saturation.high}
              ∪ (primary_diagnosis ∈ {capacity_bottleneck} 时取 by_turn TOP 饱和转向)
```

**触发条件**（OR，任一成立才查溯源，避免叙事发散）：
```
primary_diagnosis.type == "capacity_bottleneck"
  or "spillback" in detected_problems
  or turn_saturation_max ≥ saturation.oversaturation
```
`basically_matched` / 纯 `timing_optimizable` 不触发。

### 2.2 思考过程（diagnosis_story）

`problem_evidence_service._build_diagnosis_story()` 在 `corridor` beat 后追加：

## 8. 实现完成纪要（2026-06-29）

**状态：P0–P3 全部完成，`bash scripts/regression.sh` 全绿（后端 188 + 前端 134）。**

### 8.1 关键设计决定

1. **演示路口保留 `011wwe28ctu00001`（奥体西路×经十路）**：实测分向饱和 8 条、过饱和 4、最大 1.73、一跳来源 99，问题转向溯源集中（东进口左转 81.8% 途经岔口东南直行）。此前"缺 DWS 覆盖"的判断已过时。
2. **`FlowTraceService` 自取数**（与 timing/corridor 一致），在 orchestrator `data_fetch` 之后、`problem_evidence` 之前构建，结果写 `data_payload["flow_trace"]`，供 diagnosis_story / action_plan / 地图 / meta 共用。
3. **一跳锁定**：同 (本进口方位, 本转向, 上游方位, 上游转向) 取 `flow_share_ratio` 最大；再按上游方位归并、coverage 降序。
4. **来源结构**：`coverage_high=70 / gap_significant=25`（`thresholds_extensions.yaml: flow_trace`）判 single/multi/local。
5. **治理双通道**：
   - **能力瓶颈**（单点无解）→ `action_plan.action_type = upstream_coordination`，正文落脚上游来源路口。
   - **配时可优化等其它主诊断** → 主方案不变，新增 `flow_timing_governance.flow_trace_supplement`「流量溯源补充」叠加上游协同维度（演示路口即此路径）。
6. **口径红线**：途经率≠占比，全链路文案用「途经/来自」，禁「贡献/占比」；统一标注「近月同时段规律」。

### 8.2 四链路落点（问题诊断环节）

| 链路 | 落点 |
