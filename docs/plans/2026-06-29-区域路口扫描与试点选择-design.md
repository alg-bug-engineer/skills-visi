# 区域路口扫描与试点选择 · 设计文档

> 日期：2026-06-29 ｜ 状态：已确认设计，待实现
> 目标读者：交通智能体研发；面向"领导审批更多落地路口"的决策支撑

## 1. 背景与目标

### 1.1 要解决的真问题
智能体已能基于真实数据对**单个路口**做认知—感知—诊断，并给出信控治理建议。但"假定采纳建议后能带来什么效果"在**没有历史前后数据、没有领域专家配合**的前提下很难直接评价。

经过讨论收敛到一个**当下可立即交付、且直接服务于审批决策**的方向：

> 不去硬证"单点治理效果"，而是把智能体的**诊断认知能力规模化到全区域所有路口**，
> 生成一张"全域体检地图"，清晰呈现每个路口的认知情况与问题严重程度，
> 并据此筛选出**配时可解、改善潜力高**的路口作为试点候选 —— 让领导先看到"哪些路口值得投、哪些不值得投"，把审批从"逐路口拍脑袋"变成"看榜单选试点"。

### 1.2 核心交付物
1. **区域批量扫描引擎**：对全部信号路口逐路口跑完整诊断（信控视角），产出快照。
2. **扫描快照存储**：可复现、可对比、可秒级读取。
3. **扫描 API**：按时段/指标过滤、路口详情。
4. **高德地图试点选择前端**：全域路口着色地图 + 过滤 + 路口详情面板 + **试点推荐榜**。

### 1.3 明确不做（YAGNI / 本期边界）
- 不做"单点治理前后效果"的真实落地实测（无历史数据，另案）。
- 不做配时方案自动下发 / 仿真验证（后续阶段）。
- 不依赖任何领域专家人工标注。
- v1 不做 5 分钟级地图渲染（时段级足够支撑选点决策）。

## 2. 关键设计决策（已与需求方确认）

| # | 决策点 | 结论 | 理由 |
|---|--------|------|------|
| 1 | 区域范围 | **全部信号路口**（当前版本 `dim_inter_info` 中 `is_signalized=1` 且有 `geom_center`） | 体现"全域体检"的规模感，再用筛选收敛到试点 |
| 2 | 复用方式 | 新建独立文件夹 `region-scan/`，Python 侧**复用现有 `intersection_agent` 服务**做认知与诊断；**结合 inter 路口诊断技能，信控视角优先** | 不重复造 DB / 评分 / 诊断逻辑 |
| 3 | 分层口径 | **逐路口跑完整诊断**（非廉价批量近似） | 需求方要保真；离线批处理 + 快照，性能可接受 |

### 3.1 复用的现有能力（不重写）
- `intersection_agent.db.postgres.PostgresPool` —— 异步 PG 访问。
- `IntersectionCognitionService.fetch()` —— 单路口认知 + 进口道指标。
- `ProblemEvidenceService.build()` —— 证据链。
- `RuleEngine.diagnose_comprehensive()` —— 规则诊断（无 LLM）。
- `FlowTimingGovernanceService.build()` —— **信控视角**问题检测 + 治理建议 + problem_source/severity。
- `utils.corridor_metrics`（`congestion_score` / `severity_level` / `level_label`）—— 着色与评分口径，与现有干线扫描一致。
- `utils.data_window.build_data_window()` —— 时段→`day_of_week`/`step_index` 映射（早高峰/平峰/晚高峰）。
- 坐标：`geom_center` 为 WKT `POINT(lon lat)`，沿用现有干线扫描在高德上的渲染约定。

## 4. 数据流与分层口径

### 4.1 时间粒度（v1）
工作日三时段：早高峰 / 白平峰 / 晚高峰（复用 `build_data_window`）。每路口在每个时段各跑一次诊断。

### 4.2 路口分层（核心分析价值）
对每个路口、每个时段，由完整诊断结果导出 `problem_band`：

- **配时可解**：失衡 / 空放 / 绿灯利用率低 / 相位结构问题为主，且信控改善上限高 → 试点首选。
- **工程可解**：过饱和、需求 > 通行能力，信控改善上限低 → 标注"配时救不了，别浪费投入"。
- **无明显问题**：运行平稳。
- **数据不足**：该时段缺运行数据，单独标注，不参与排序。

> `problem_band` 由 `FlowTimingGovernanceService` 的 `match_verdict` / 检出问题类型 +
> 饱和度阈值 + 信控改善上限综合判定（具体映射在实现计划中细化并写测试）。

### 4.3 试点推荐分
`pilot_score = severity_weight × control_improvement_ceiling × data_confidence`
仅对 `problem_band=配时可解` 的路口计算并排序，产出**试点推荐榜**。

## 5. 扫描快照数据模型

`scan_run`：`run_id`、`created_at`、`region`、`version_id`、`periods[]`、`status`、`intersection_total`、`covered`。

`intersection_scan`（每条 = 路口 × 时段）：
- 标识：`run_id`、`inter_id`、`inter_name`、`lon`、`lat`、`period`
- 认知：`scene_type`、`pressure_level`、关键进口道画像摘要
- 指标：`saturation_max`、`unbalance_index`、`green_utilization`、`queue`、`delay`
- 诊断：`top_issues[]`、`severity`、`cause_scores`、`control_improvement_ceiling`
- 分层：`problem_band`、`pilot_score`
- 信控建议：`governance_summary`、`governance_actions[]`
- 质量：`has_data`、`data_quality_tags[]`、`uncertainty[]`

存储选型：优先 PG 新 schema（如 `scan`）便于查询与多次对比；离线快照也落 JSON 便于前端直读与归档。具体在实现计划确定。

## 6. 扫描 API（草案）
- `POST /api/scan/runs` —— 触发一次区域扫描（参数：region、periods）。
- `GET  /api/scan/runs` —— 列出历史扫描。
- `GET  /api/scan/runs/{run_id}?period=&band=&metric=` —— 地图所需的路口集合（含坐标、着色值、分层）。
- `GET  /api/scan/runs/{run_id}/pilots` —— 试点推荐榜。
- `GET  /api/scan/intersections/{inter_id}?run_id=&period=` —— 单路口完整认知+诊断+信控建议详情。

## 7. 前端（高德试点选择地图）
- 技术栈：Vite + Vue 3（与 `frontend-v2` 一致），高德 JS API（key 见 `docs/gd-js20.md`）。
- 地图层：全域路口标记，按"时段 × 指标/分层"着色（饱和度热度 / problem_band 色板）。
- 控件：时段切换、指标切换、分层过滤、严重度阈值。
- 交互：点击路口 → 右侧面板展示六维认知摘要 + 诊断结论 + **信控治理建议** + 试点推荐理由。
- **试点推荐榜**：侧栏列表，配时可解路口按 `pilot_score` 排序，点击联动地图定位。
- 图例与"为何不选"说明：过饱和路口明确标注"工程可解、配时无效"，体现"替领导挡掉无效投入"。

## 8. 性能与运行方式
- 扫描为**离线批处理**：全部信号路口 × 3 时段 × 完整诊断，预计数分钟至数十分钟，产出快照。
- 地图只读快照，秒级响应。
- 批处理具备断点续跑、单路口失败隔离（失败计入 `data_quality_tags`，不中断整体）。
- 并发：受 `PostgresPool` 限制，控制并发度，避免压垮 PG。

## 9. 风险与缓解
| 风险 | 缓解 |
|------|------|
| 坐标系（GCJ-02/WGS84）导致地图偏移 | 沿用现有干线扫描渲染约定；实现时用已知路口比对校验 |
| 全量逐路口诊断耗时 | 离线批处理 + 快照 + 断点续跑；并发可调 |
| 部分路口缺时段数据 | `has_data=false` 单独标注，不参与排序，不污染榜单 |
| `problem_band` 口径误判 | 口径集中在一处 + 单元测试 + 抽样人工核对几例 |
| 与现有 backend 服务签名耦合 | 仅通过服务公开方法调用；签名变化由回归测试兜底 |

## 10. 验收标准（v1）
1. 一条命令完成全区域扫描并生成快照。
2. 高德地图展示全部有坐标的信号路口，按时段/指标/分层着色正确。
3. 点击任一路口可见其完整认知 + 信控诊断 + 治理建议。
4. 试点推荐榜列出配时可解路口并按潜力排序，可与地图联动。
5. 过饱和路口被正确标注为"工程可解 / 配时无效"。
6. 关键分层与评分逻辑有单元测试。
