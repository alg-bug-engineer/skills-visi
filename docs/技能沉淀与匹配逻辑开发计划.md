# 技能沉淀与匹配逻辑 — 开发计划

> 版本：2026-06-26  
> 状态：已实施（2026-06-26）

## 1. 背景与目标

路口诊断智能体支持将一次完整诊断（规则命中、数据查询规格、用户约束、治理公式）固化为 **路口 Skill 包**。当用户再次询问同一路口时，应优先复用历史沉淀经验，同时在文案与交互上明确体现「已有技能、正在复用」。

本次改造聚焦三件事：

1. **标签体系**：为 Skill 建立可检索、可扩展的标签（路口、时段、问题、方向等），支撑长期规划中的技能库检索与相似匹配。
2. **命中提示**：匹配到历史 Skill 时，在 SSE 步骤与诊断文案中提示用户，体现经验复用。
3. **约束感知的匹配与二次确认**：用户本次约束与历史 Skill 不一致时，不走快路径；一致或无约束时走快路径，但治理建议仍保留二次确认，且仅在用户**补充新经验**时才触发 Skill 固化确认。

## 3. 匹配逻辑

### 3.1 匹配维度（AND）

```
inter_id 相同
AND problem_type 相同
AND time_period_label 相同
AND directions 兼容（Skill 无方向标签时跳过，兼容旧包）
AND 用户约束兼容（见 3.2）
```

### 3.2 用户约束兼容规则

| 用户本次输入 | 历史 Skill 约束 | 结果 |
|--------------|-----------------|------|
| 无 `user_suggestion` | 任意 | ✅ 匹配（复用 Skill，诊断时注入历史约束） |
| 有约束 A | 无约束 | ❌ 不匹配 → 全量重算 |
| 有约束 A | 有约束 B，A≠B（归一化后） | ❌ 不匹配 → 全量重算 |
| 有约束 A | 有约束 A | ✅ 匹配 |

**归一化**：去首尾空白、合并连续空白、全角标点统一，用于文本相等比较。

**合理性说明**：

- 用户提出与历史不同的约束（如「绿灯不超过 10 秒」vs 历史「可延长 20 秒」），代表治理边界已变化，旧 Skill 快照不再可信，必须重新查数、重跑规则与约束量化。
- 用户未提约束时，历史 Skill 的约束是工程师已确认的经验，可在生成建议时注入，避免重复输入。
- 仅当用户在**二次确认阶段补充**新经验时，才视为需要更新 Skill，避免每次复用都弹固化确认。

### 3.3 匹配结果结构

```python
@dataclass
class SkillMatchResult:
    skill: SkillRecord | None
    matched: bool
    reason: str  # matched | no_skill | direction_mismatch | constraint_mismatch
    detail: str | None  # 用户可见说明
```

## 4. 会话流程改造

### 4.1 状态字段

`Session` 新增：

- `skill_reuse_mode: bool` — 当前会话是否处于「历史 Skill 复用」分支

### 4.2 流程对比

#### 原快路径（改造前）

```
命中 Skill → 查数+规则 → 直接生成建议 → 校验一致则 verified 结束
```

#### 新快路径（改造后）

```
命中 Skill → 查数+规则（带 skill rule_ids）
  → 提示「发现沉淀技能」
  → 若无用户约束则注入 Skill 历史约束
  → awaiting_confirm(suggestion_action=awaiting_generate)  【保留二次确认】
  → 用户确认：
       · 仅「是」/无补充 → 生成建议 → done（skill_action=reused_no_persist，不固化）
       · 补充新经验   → 生成建议 → awaiting_confirm(skill_action=awaiting_create/更新)
```

#### 约束不匹配

```
find_match → constraint_mismatch
  → SSE 提示「约束与历史技能不一致，将完整重新诊断」
  → 不走 skill_fast_path，全量 pipeline
  → 后续按普通流程（有约束则治理后固化确认）
```

### 4.3 meta 字段约定

| 字段 | 含义 |
|------|------|
| `skill_match_reason` | matched / constraint_mismatch / direction_mismatch / no_skill |
| `skill_reused` | bool，本次是否基于历史 Skill 诊断 |
| `skill_action=reused_no_persist` | 复用 Skill 生成建议，用户未补充经验，未固化 |
| `skill_action=skipped_no_user_suggestion` | 普通流程，无约束不固化（保持不变） |

## 5. 前端交互

### 5.1 SSE `skill_match` 步骤

- `matched=true`：在「匹配路口」步骤追加  
  `📚 发现沉淀技能：{路口} · {时段} · {方向}，将基于历史经验辅助诊断`
- `matched=false, reason=constraint_mismatch`：  
  `用户约束与历史技能不一致，将重新完整诊断`

### 5.2 诊断确认文案

后端在 `skill_reuse_mode` 下，问题确认消息顶部插入沉淀技能说明块。

### 5.3 技能固化步骤

- `skill_action=reused_no_persist`：不展示「技能固化」确认步骤
- `skill_action=awaiting_create`：保持现有 SkillBuildOverlay 流程

## 7. 测试用例

1. **标签写入**：固化后 `skill.meta.json` 含 `tags.directions`
2. **无约束复用**：同路口二次提问 → `skill_fast_path` + `awaiting_generate` → 确认「是」→ `reused_no_persist`
3. **约束冲突**：历史 Skill 约束 A，用户输入约束 B → 不匹配，全量诊断
4. **约束一致复用**：用户输入与历史相同约束 → 匹配 → 二次确认仅「是」→ 不固化
5. **复用后补充经验**：二次确认时补充新约束 → 生成建议 + `awaiting_create`
6. **方向不匹配**：Skill 为南北向，用户问东西向 → 不匹配（若 Skill 有方向标签）

## 8. 风险与回滚

- **旧 Skill 无 tags**：从 `inter_id` / `time_period_label` 等回填，方向检查跳过
- **快路径行为变化**：集成测试 `test_fast_path_skips_confirm_when_verified` 需改为新流程断言
- 回滚：保留 `find_match` 旧签名，可通过配置开关 `SKILL_REUSE_V2=0` 恢复（本次不实现开关，直接切换）
