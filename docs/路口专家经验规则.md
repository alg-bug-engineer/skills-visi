# 路口场景认知与问题诊断 · 专家经验规则

> **文档定位**：本文档集中收录路口 **场景认知** 与 **问题诊断** 两阶段的领域专家经验，供智能体推理、规则编写与人工复核使用。  
> **机器可读真源**（阈值、检查单、场景分化）仍以 YAML 为准，本文与之对齐并补充定性判别逻辑。  
> **依据来源**：`scene-cognition/references/rules.md`、`problem-diagnosis/references/rules.md`、两阶段检查单规则及 `docs/算法设计文档/交通智能体问题检查单-0614.docx`。

---

## 1. 总体方法论

### 1.1 两阶段分工

| 阶段 | 职责 | 不做的事 |
| --- | --- | --- |
| **场景认知** | 事实抽取、指标计算、质量标记、场景标签；构建「供给—需求—状态」证据底座 | 最终问题归因、策略动作、配时秒数 |
| **问题诊断** | 将画像转化为可排序问题集合，区分主因维度与信控改善空间 | 把结构性/秩序问题伪装成配时问题 |

场景认知输出的是 **可引用证据**（含 `checklist_item_id`、`fact_id`）；问题诊断通过 `profile_evidence_ref` 回链，不得脱离证据臆造结论。

### 1.2 问题三分法

诊断结论必须区分：

1. **信控可改善问题** — 可通过周期、绿信比、相位相序、协调、触发控制等手段改善。
2. **工程或秩序治理为主问题** — 渠化、出入口、公交停靠、施工占道、违停等约束主导，信控只能缓解。
3. **数据不确定问题** — 关键证据缺失或冲突，需补采后再下结论。

### 1.3 「城市交通 CT 体检」思维

以 **供给—需求—状态** 三维画像作为事实底座：

1. 先描述道路、设施和信号 **供给**；
2. 再描述交通需求的总量、空间、时间和方式 **结构**；
3. 最后描述运行状态、拥堵阶段和风险 **扩散特征**。

缺少关键证据时输出补采建议，不让后续阶段静默推理。

## 2. 场景认知专家经验

### 2.1 供给画像

**应覆盖的要素**

- 进口道、出口道、车道数、车道功能、渠化、转向车道长度
- 路口面积、相邻路口间距
- 行人和非机动车过街设施
- 信号机能力、当前周期、相位相序、绿信比分配、最小安全约束

**结构性瓶颈线索**（写入 `supply_profile`，**不直接归因配时**）

| 线索 | 专家含义 |
| --- | --- |
| 三进两出、出口道少于进口道 | 漏斗效应，进口排队易外溢 |
| 短车道排队溢出 | 存储空间不足，信控防锁死优先 |
| 相邻路口间距 < 150 m | 短间距串联，协调/溢流传导风险高 |
| 转弯半径过小、路口面积过大 | 清空与启动损失偏高，信控效率上限低 |

**环境上下文**（写入 `context_tags`）

对学校、医院、商圈、景区、交通枢纽、桥隧匝道、施工点等强吸引或强约束对象，应说明其如何改变需求峰值、慢行冲突、停车绕行或排队存储。

### 2.2 需求画像

**三维需求分析**

| 维度 | 关注内容 |
| --- | --- |
| **总量** | 高峰小时流量、机动车出行强度、过境流量、在途量 |
| **空间** | 进口、车道、转向、相位、行人/非机动车分布 |
| **时间** | 早晚高峰、平峰、夜间、周末、节假日、特殊事件 |

**出行目的与形态**

| 目的类型 | 典型特征 |
| --- | --- |
| 刚性通勤 | 潮汐明显、集中到达 |
| 学校 / 医院 | 短时高密度停车、过街、临停 |
| 景区 / 商圈 | 外地车、找位绕行、慢行集散 |
| 货运 / 公交接驳 | 停靠占道、波峰与主线干扰 |

**可调控需求**：具备错峰、公交慢行转移、外围停车换乘或绕行可能的需求，可作为后续策略阶段的需求管理线索（标明时间/方式/空间弹性）。

### 2.3 状态画像

**饱和度不能单独定拥堵**

`saturation` 判断供需压力，须结合以下指标共同解读：

- `avg_delay_s`、`queue_m`、`queue_storage_ratio`
- `spillback_risk`、`stop_count`
- `green_utilization`、`empty_green_rate`

**拥堵空间形态**

| 形态 | 判定线索 |
| --- | --- |
| **点状** | 单一路口排队 |
| **线状** | 连续路段排队、绿波断裂 |
| **面状** | 多个相邻片区扩散；`imbalance_index` 高值时段辅助 |

**拥堵发展阶段**

| 阶段 | 判定线索 |
| --- | --- |
| **萌芽** | 排队接近存储空间、速度开始下降 |
| **扩散** | 排队传导至相邻路口或下游余量持续下降 |
| **消散** | 延误、排队、溢流风险连续回落 |

**反常组合（优先怀疑非配时主因）**

低饱和度 + 高延误/长排队 → 优先怀疑：

- 数据质量异常
- 下游阻塞
- 停车占道、公交停靠
- 行人/非机动车冲突
- 渠化问题

应写入 `quality_tags` 或 `uncertainty`，**不强行构造过饱和结论**。

### 2.4 压力等级与指标口径

- `pressure_level` 取饱和度、延误、排队、溢流、冲突风险中的 **最大压力**。
- `saturation = volume / capacity`；无能力值时不臆造，标记缺少 `metrics.capacity`。
- `queue_storage_ratio = queue_m / storage_m`，判断排队是否接近进口存储空间。
- `movement_saturation` 仅在分流向流量与分流向能力 **同时存在** 时计算。
- `green_utilization` 与 `empty_green_rate` 同时存在时都保留；二者冲突时写入 `quality_tags`。

### 2.5 关键指标解读原则

基于 5 分钟 DWS **逐步时序**（`step_index`），不得凭峰值聚合反推具体时段。

**解读侧重**（有数据且有意义时）

- 饱和/过饱和时段、排队延误、绿灯利用
- 流量结构、失衡与服务水平
- 设施约束与群众反馈、配时方案划分

**边界**

- 解读阶段 **不做问题归因**，**不输出配时调整建议**。
- 若全天均未达高饱和阈值，应明确说明运行总体平稳，可补充峰值时刻与相对繁忙方向。
- 缺少 DWS 时序时，说明无法推断时段分布，并指向 `inter_evaluation` / `turn_saturation` 补采。

### 2.6 数据质量红线

以下情况必须写入 `quality_tags` 和 `validation_errors`：

- 缺少 `scope.level`、`metrics.volume`、`metrics.capacity`、`metrics.avg_delay_s` 等关键字段
- `capacity <= 0` 但存在流量
- 检测器在线率 < 0.9
- 数据延迟 > 120 秒
- 灯态、相位或配时字段缺失，无法建立控制画像
- 指标明显冲突（如低饱和 + 高延误/长排队）
- 目标路口无 DWS 覆盖 → 标记 `missing_dws_coverage`

## 3. 问题诊断专家经验

### 3.1 主因维度判别

诊断前先判断问题主因属于哪一类：

| 主因类型 | 典型线索 | 信控角色 |
| --- | --- | --- |
| **静态供给短板** | 渠化不匹配、存储不足、路口过大、短间距 | 优化上限低，需说明非信控治理 |
| **动态供需失衡** | 高峰超能力、节假日激增、潮汐未适配 | 中到高，受供给上限约束 |
| **信控参数不适配** | 失衡、空放、周期不合理、方案粗糙 | 高杠杆 |
| **交通秩序干扰** | 违停、临停、加塞、非机动车违法 | 只能配合缓解 |
| **外部事件** | 事故、施工、恶劣天气、大型活动 | 应急/降级策略 |
| **数据不确定** | 关键字段缺失或冲突 | 降级，输出补采 |

**判定原则**：只有当问题能通过周期、绿信比、相位相序、协调、触发控制或边界控流改善时，才判为信控可改善问题。

### 3.2 静态短板专家线索

**区域维度**（结构性 / 需求管理问题）

- 路网密度 < 8 km/km²、支路微循环缺失
- 职住分离明显、公共交通覆盖不足
- 停车供给短缺、轨道/公交接驳弱

**路段维度**（道路组织与秩序治理）

- 公交站点过密（< 300 m）或过疏（> 800 m）
- 单位出入口密集、强吸引点出入口排队外溢
- 缺少非机动车专用道、短间距无信号过街频繁
- 断面设计不合理

**路口维度**（信控上限须明示）

- 车道功能与转向流量不匹配
- 左转/右转专用车道长度不足（< 50 m）
- 路口面积 ≥ 2000 m²
- 进口/出口车道数不匹配（漏斗效应）
- 相邻路口间距 < 150 m
- 行人/非机动车等待区不足

→ 不得把上述结构性问题 **完全归为配时问题**。

### 3.3 动态供需与秩序线索

**动态供需失衡**

- 高峰小时流量接近或超过通行能力
- 节假日、开学日需求激增
- 强吸引点停车排队占道
- 可变/潮汐车道未按主流向启用

**交通秩序问题**（信控只能配合）

- 路内违停、双排停车、网约车/快递临停
- 违规变道加塞
- 非机动车闯红灯或逆行
- 公交停靠阻塞主线

**外部事件主导**

- 事故、故障车、施工、恶劣天气、大型活动、临时管制
- 通行能力突降或需求突变 → 触发应急或降级类策略

### 3.4 信控问题专家判别

#### 路口失衡

- **现象**：各进口/流向/相位饱和度差异大，部分方向持续排队，其他方向空闲
- **量化**：`imbalance_index ≥ 0.30`、相位饱和度极差 `≥ 0.25`、车道利用率离散、关键相位排队不清空
- **问题码**：`service_imbalance`
- **信控杠杆**：高

#### 溢流或锁死风险

- **现象**：下游排队回溢、出口无余量、短车道排队进入相邻车道、相邻路口排队互相影响
- **量化**：`spillback_risk ≥ 0.80`、`queue_storage_ratio ≥ 0.80`、连续不清空周期 `≥ 3`
- **问题码**：`spillback`
- **信控杠杆**：高；**优先防锁死**

#### 空放 / 绿灯损失

- **现象**：某相位绿灯期间无车或少车通过；平峰/夜间沿用高峰方案导致无效等待
- **量化**：`empty_green_rate ≥ 0.20`、`green_utilization < 0.50`、启动损失 `≥ 2 s`
- **问题码**：`empty_green`
- **信控杠杆**：高

#### 相位相序不合理

- **现象**：高冲突流向同相放行；左转专用车道无左转相位；全红清空与实际冲突不匹配；相序增加启动损失
- **归类**：安全或效率问题；涉及冲突时 **安全优先**
- **问题码**：`phase_sequence_conflict`、`phase_channel_mismatch`

#### 行人/非机动车配时不足

- **现象**：过街时间不足；右转与行人/非机动车冲突高；二次过街无安全岛；学校/医院/商圈慢行需求高
- **输出**：慢行保护建议 + 人工复核
- **问题码**：`pedestrian_protection_gap`

#### 干线协调问题

- **现象**：连续停车、绿波通过率低、协调方向与主车流不匹配、协调速度偏离实际、协调周期/范围不当
- **归类**：走廊协调问题，**不是单点配时问题**
- **量化**：`green_wave_pass_rate < 0.55`、`offset_deviation_s ≥ 10`
- **问题码**：`green_wave_break`

### 3.5 问题码与触发条件速查

| 问题码 | 中文名 | 典型触发条件 | 信控改善空间 |
| --- | --- | --- | --- |
| `phase_sequence_conflict` | 相位相序/冲突风险 | `conflict_risk ≥ 0.60`、直左/机非/人车冲突标签 | 中—高，需安全校验 |
| `spillback` | 排队溢出/锁死 | `spillback_risk ≥ 0.80`、`queue_storage_ratio ≥ 0.80`、连续周期不清空 | 高，优先防锁死 |
| `oversaturation` | 过饱和 | `saturation ≥ 0.90`、连续不清空 `≥ 3` | 中—高，受供给上限影响 |
| `service_imbalance` | 服务失衡 | `imbalance_index ≥ 0.30`、相位饱和度极差大 | 高 |
| `excess_delay` | 延误/停车偏高 | `avg_delay_s ≥ 60`、`stop_count ≥ 2.5` | 中—高 |
| `empty_green` | 空放/绿灯损失 | `empty_green_rate ≥ 0.20`、`green_utilization < 0.50` | 高 |
| `lane_mismatch` | 渠化不匹配 | `lane_utilization_cv ≥ 0.35`、短转向车道、漏斗效应 | 低—中，常需渠化 |
| `manual_intervention` | 人工干预频繁 | 人工接管次数 ≥ 3 或相关标签 | 中 |
| `plan_granularity` | 方案精细度不足 | 时段方案 ≤ 5 套、特殊日未覆盖、时段内波动大 | 中 |
| `green_wave_break` | 上下游协调/绿波断裂 | 绿波通过率低、相位差偏差大、相邻排队传导 | 中—高 |
| `external_disturbance` | 外部干扰主导 | 违停、公交停靠、施工占道、事故等 | 低，信控仅缓解 |
| `public_complaint` | 投诉/专项保障 | 投诉台账或现场调研非空 | 中，需人工审核 |
| `phase_channel_mismatch` | 相位与渠化不匹配 | 专用车道无对应相位 | 低，常需渠化 |
| `oversized_intersection` | 路口面积过大 | `intersection_area_m2 ≥ 2000` | 低 |
| `cycle_timing_issue` | 周期过长/过短 | `cycle > 190 s` 或 `cycle < 60 s` | 中 |
| `downstream_blockage` | 下游阻塞 | 溢流 + 施工/事故/出入口标签 | 中 |
| `pedestrian_protection_gap` | 慢行保护不足 | 学校/医院 POI、相关投诉 | 中 |
| `stable` | 运行基本稳定 | 无明显问题触发 | 低 |

### 3.6 诊断优先级

排序原则：**先看安全和扩散风险，再看效率和体验**。

| 优先级 | 类别 | 典型问题 |
| --- | --- | --- |
| 1 | **安全** | 相位冲突、机非/人车冲突、恶劣天气安全裕度不足 |
| 2 | **扩散** | 溢流、锁死、下游阻塞、连续周期不清空 |
| 3 | **供需** | 过饱和、关键进口持续长排队 |
| 4 | **效率** | 服务失衡、空放、周期不合理 |
| 5 | **结构** | 渠化不匹配、方案精细度不足 |
| 6 | **体验** | 外部干扰、投诉、专项保障诉求 |

### 3.7 信控优化上限判定

| 条件 | `control_improvement_ceiling` |
| --- | --- |
| 结构性短板 ≥ 2 项且无动态高杠杆问题 | `low` |
| 动态失衡 / 空放 / 周期问题为主 | `high` |
| 外部干扰或秩序问题主导 | `none` 或 `low` |
| 数据关键字段缺失 | 降级并输出 `uncertainty` |

区域治理场景：静态短板 ≥ 2 且无高杠杆动态问题时，上限应为 `low` 或 `none`。  
应急场景：`problem_source` 优先判为外部干扰，不得将设施问题伪装成配时问题。

### 3.8 成因六维归因

问题命中后，按以下维度累计主因贡献（0—1 归一，输出 `cause_scores`）：

| 维度 | 含义 | 典型关联问题码 |
| --- | --- | --- |
| **supply** 供给 | 渠化、存储、路口几何 | `lane_mismatch`、`phase_channel_mismatch`、`oversized_intersection` |
| **demand** 需求 | 流量超能力、潮汐 | `oversaturation` |
| **control** 控制 | 配时、相序、协调 | `service_imbalance`、`empty_green`、`spillback`、`green_wave_break`、`cycle_timing_issue` |
| **order** 秩序 | 违停、临停、违法 | `external_disturbance`、`public_complaint` |
| **event** 事件 | 事故、施工 | `downstream_blockage` |
| **data_quality** 数据 | 缺测、冲突 | 各问题置信度降级 |

## 4. 按场景类型分化规则

`scene_type` 由 `scene_type_rules.yaml` 解析；诊断阶段由 `classify_diagnosis_context.py` 应用权重与上限。

| scene_type | 诊断侧重 | 优先提升权重 | 降低权重 | 默认信控上限 |
| --- | --- | --- | --- | --- |
| **单点配时优化** | 失衡、空放、过饱和 | `service_imbalance`、`empty_green`、`oversaturation` | `green_wave_break` | high |
| **干线协调优化** | 绿波断裂、溢流传导 | `green_wave_break`、`spillback` | `plan_granularity` | medium |
| **区域拥堵治理** | 过饱和、下游阻塞 | `oversaturation`、`spillback`、`downstream_blockage` | `empty_green`、`cycle_timing_issue` | low |
| **应急优先** | 溢流、外部干扰 | `spillback`、`downstream_blockage`、`external_disturbance` | `plan_granularity`、`empty_green` | medium |

**问题来源（problem_source）辅助规则**

| 条件 | problem_source | 信控上限 |
| --- | --- | --- |
| 静态触发 ≥ 2 且无动态高杠杆 | `static_supply_constraint` | low |
| 外部干扰/下游阻塞 + 施工/事故/违停标签 | `external_disturbance` | none |
| 秩序类问题 + 秩序维度得分最高 | `traffic_order_interference` | low |
| 动态触发 ≥ 1 且控制维度 ≥ 0.35 | `control_parameter_mismatch` | high |
| 无触发问题 | `stable_operation` | none |

## 5. 统一阈值速查

> 完整定义与注释见 `thresholds.yaml`（`load_config.threshold("分组.键名")`）。

### 5.1 场景认知常用阈值

| 键 | 值 | 用途 |
| --- | --- | --- |
| `saturation.high` | 0.80 | 高饱和时段窗口 |

## 6. 检查单专家要点

检查单逐项定义见各阶段 `checklist_rules.md` 与 YAML 真源；以下为专家视角的 **必查逻辑**，非表映射。

### 6.1 场景认知检查单

**检索顺序**：静态供给 → 动态运行 → 控制画像 → 环境上下文。不得跳过检查单臆造画像字段。

| 类别 | 专家关注点 |
| --- | --- |
| 静态供给 | 道路等级与形态、渠化与车道明细、相邻间距（取各进口 link 长度最小值） |
| 动态运行 | 转向/车道流量、饱和度、排队延误停车、绿灯利用、车道能力 |
| 控制画像 | 当前方案与周期、阶段绿信比、信号—车道映射、时段方案套数 |
| 环境上下文 | 投诉台账、现场调研 |

动态 DWS 表按 `day_of_week` 拉取全天 5 分钟时序再聚合；缺 DWS 时必须标记 `missing_dws_coverage`。

### 6.2 问题诊断检查单

**检查顺序**：静态 → 动态 → 特殊需求 → 成因汇总。

| 类别 | 专家关注点 |
| --- | --- |
| 静态 | 相位—渠化匹配、流量—渠化匹配、转向存储、漏斗效应、机非冲突、路口面积、相邻间距 |
| 动态 | 过饱和、溢流、下游阻塞、失衡、空放、周期、延误、方案精细度 |
| 特殊 | 学校/医院/公交/施工 POI、投诉与调研 |
| 汇总 | 六维 `cause_scores` |

每项状态：`passed` / `triggered` / `no_data` / `error` / `skipped`；缺少关键画像字段时标记 `no_data` 并在 `uncertainty` 说明。

## 7. 不确定性处理

以下情况必须写入 `uncertainty` 或 `validation_errors`：

**场景认知**

- 关键字段缺失、检测器离线、数据延迟、灯态/配时缺失
- 指标互相冲突
- 无 DWS 时序，无法推断时段分布

**问题诊断**

- 缺少流量、能力、延误、排队、相位、车道利用率等关键字段
- 低饱和 + 高排队/高延误长期并存
- 外部事件主导但持续性、空间范围或责任边界不清
- 冲突仅有文本标签，无轨迹或冲突点证据
